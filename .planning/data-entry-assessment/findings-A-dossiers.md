# Findings — Slice A: Dossier creation & edit forms (all 8 types)

**Scope:** create wizard + per-type config/schema/steps for `country`, `organization`,
`forum`, `engagement`, `topic`, `working_group`, `person`, `elected_official`; the
`dossiers-create` / `dossiers-update` Edge Functions; and the absence of edit/inline-edit UI.
Verified against the **live staging schema** (`zkrcjzdemdmwhearhfgg`) via read-only SQL.

**Summary.** Dossier data entry is **create-only** and funnels every type through one
Edge Function (`dossiers-create`) that correctly maps each type to its extension table
(`engagement → engagement_dossiers` ✓, `country → countries` ✓, `person → persons` ✓) and
atomically rolls back the base row if the extension insert fails. The create _table/column_
mapping is largely sound. The real problems are structural: (1) **no edit capability exists
for any of the 8 types** — the update hooks are exported but wired to zero components, and the
`dossiers-update` Edge Function is contract-drifted and references columns that don't exist on
`dossiers`; (2) a **systemic client-side validation gap** — the wizard's Complete button gates
only on name length, so every per-type "required" Zod rule is unenforced at submit, letting
incomplete records persist silently or surface as confusing 500s; (3) **silent loss of
engagement participants**. RTL/a11y and i18n-namespace registration are in good shape.

## Verified correct (checked, not defects)

- **Extension-table routing** in `dossiers-create/index.ts:296-304` is correct for all types,
  including the historically-wrong `engagement → engagement_dossiers` (NOT `engagements`) and
  `country → countries`. Extension insert is atomic (rolls back the base dossier on failure,
  `:322-352`).
- **Engagement enums** in `engagement.schema.ts:5-26` (`bilateral_meeting`/… and
  `diplomatic`/…) match the live `engagement_dossiers` CHECK values — the old frontend-only
  invented enums are gone.
- **NOT-NULL extension columns without server validation are safe** where the DB supplies a
  default: `engagement_dossiers.timezone` (`'Asia/Riyadh'`), `.engagement_status` (`'planned'`),
  `.lifecycle_stage` (`'intake'`), `working_groups.wg_status` (`'active'`). `topics.theme_category`
  has **no** default + a CHECK, so the A-3 gap can produce a 500 there (noted in A-3).
- **i18n**: every wizard namespace (`form-wizard`, `country-wizard`, …, `elected-official-wizard`)
  plus `dossier`/`validation`/`contextual-help` is registered in `frontend/src/i18n/index.ts`
  (no silent-EN-fallback). Person/AR inputs use `dir={direction}`/`dir="rtl"`; layouts use logical
  props (`ms-/me-/text-start`) — no RTL violations found in the wizard.
- **`forum_type` silent-drop already fixed**: `ForumDetailsStep.tsx:1-11` documents the prior bug
  and the field was removed; `forum.config.ts` now sends only `organizing_body`, matching the
  `forums` schema (no `forum_type` column).
- **Person name composition works**: `PersonBasicInfoStep.tsx:88-100` syncs `name_en/name_ar`
  from first+last via `useEffect`, so `handleComplete` reading `values.name_en` is correct
  (the `__composed_name_*` comment in `person.config.ts` is stale but harmless).
- **work_item_dossiers linking** is not part of dossier _creation_ (work items link to dossiers,
  not the reverse) — out of scope for the create flow; no linking UI fires during dossier create.
