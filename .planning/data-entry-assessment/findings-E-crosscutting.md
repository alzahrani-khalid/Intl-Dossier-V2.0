# Findings — Slice E: Cross-cutting data-entry quality

**Scope:** Systemic data-entry quality across ~25 forms spanning settings, dossiers (wizard +
person create), work items (commitments/tasks/intake quick-forms), positions, contacts, calendar,
relationships, legislation — plus the shared form/i18n/error-handling/RTL infrastructure they all
depend on. Read-only; evidence verified against the live staging schema (`zkrcjzdemdmwhearhfgg`) and
the static i18n bundle.

**Headline:** The good news is that the **data layer is mostly honest** — almost every form fires a
real mutation against the **correct** table (`aa_commitments`, `calendar_entries`, `intake_tickets`,
`engagement_dossiers`, `email_notification_preferences` all verified correct), there are **no
phantom/stub mutations** at the persistence layer, and RTL _structural_ discipline is strong
(logical props are ESLint-enforced; `ui/**` has zero physical props; 94 forms set `dir`). The
defects cluster in four cross-cutting seams: **(1)** one form writing to a table that doesn't have
its columns (Settings — total failure), **(2)** the shared API client swallowing every server error
body, **(3)** an i18n layer where hand-rolled forms bypass the auto-translating `FormMessage` and
leak raw keys / English-in-Arabic, and **(4)** screen-reader announcement of validation errors
missing app-wide.

**Counts:** `CRITICAL: 2  HIGH: 7  MEDIUM: 16  LOW: 8`

---

## Systemic patterns (how widespread)

### S1 — The centralized API client discards every server error body (HIGH, ~22 repositories)

`frontend/src/lib/api-client.ts:60-61` (and `:88-89`) — `handleResponse` does
`throw new Error('API error ${response.status}: ${response.statusText}')` and never reads
`response.json()` on the error path. Every Edge Function returns a structured body
(`{ message_en, details: [...] }`), but **all of it is dropped**. 22 repositories/services route
through this client (work-items, persons, intake, calendar, dossiers, relationships, tags, audit,
briefings, search, import…), so any 400/422/500 surfaces to the user as a generic
`API error 400: Bad Request` — field-level and localized server messages never reach the UI. This is
the root cause behind several "generic toast on failure" sub-findings below.

### S2 — Two validation tiers; the hand-rolled tier leaks i18n (CRITICAL→MEDIUM cluster)

Forms built on the shared `Form`/`FormMessage` primitive get **free** message translation
(`frontend/src/components/ui/form.tsx:149-151` runs `t(rawBody, { defaultValue: rawBody })`), so
their `'validation:*'` zod keys resolve in both languages. Forms that hand-roll error rendering
(`{errors.x.message}`) or address a namespace with the wrong separator do **not**, and leak in both
languages. Confirmed instances: AttachmentUploader (E-2, raw dotted keys — CRITICAL),
IntakeForm (E-13), TaskEditDialog (E-11), AddDeliverableDialog (E-19), IntakeQueue (E-12),
PositionEditor/PositionDossierLinker (E-24/E-25). The classic dot-vs-colon and
string-vs-object key-collisions are the mechanism (see S3).

### S3 — Only 25 of 75 RHF forms use a zod resolver; "schema present ≠ schema enforced"

75 files import `react-hook-form`; only 25 use `zodResolver`. Some no-resolver forms are fine (wizard
_steps_ defer validation to the shell hook which does resolve), but several real forms validate
**nothing**: RelationshipForm (E-7, no resolver, no rules, error UI is dead code), and the dossier
wizard never actually _runs_ its schema as a gate (E-5). Where a schema exists, its messages are
sometimes hard-coded English (`base.schema.ts`, AddDeliverableDialog) and show English in Arabic.

### S4 — i18n bypass by hard-coded bilingual ternaries (MEDIUM, very widespread)

**1,173** occurrences of the `isRTL ? 'English' : 'العربية'` inline-string pattern across
components/pages/routes (33+ of them inside actual form files). Plus hard-coded English `placeholder=`
strings in forms (e.g. PersonCreatePage). These render _correct_ Arabic via the ternary, so they're
not "silent EN fallback" — but they live outside the bundle, are unmaintainable, and are the source
of the per-form English-in-AR leaks when the ternary is forgotten.

### S5 — Locale-unaware number/date formatting (MEDIUM)

Only **5** files use the `toFormatLocale` helper (`lib/format-locale.ts`); ~**40+** form/display
files call bare `new Intl.NumberFormat(...)` / `toLocaleString()` / `toLocaleDateString()` with no
locale or with bare `i18n.language` (which resolves to `latn` digits in Chrome). Result: Latin digits
and English date formats in the Arabic UI. ~179 raw call-sites total.

### S6 — Validation errors are visible but not _announced_ (MEDIUM a11y, app-wide)

`FormControl` correctly wires `aria-describedby` / `aria-invalid` / `aria-required`, and `FormLabel`
sets `htmlFor` — label association is **good** app-wide. But `FormMessage`
(`components/ui/form.tsx:158-167`) renders the error as a plain `<p>` with **no `role="alert"` / no
`aria-live`**, so when an error appears on submit it is never announced to screen-reader users. This
affects every RHF form. (Independently flagged by 3 of the 4 cluster audits.)

### S7 — Errors logged, not surfaced; destructive actions under-guarded (HIGH→MEDIUM)

A recurring `catch (err) { console.error(...) }` pattern swallows mutation failures with no toast and
no `role="alert"` (PositionDossierLinker E-9, PositionEditor E-24, the wizard E-5). Several
destructive actions delete immediately with native `confirm()`/`alert()` or no confirm
(PositionDossierLinker delete, CalendarEntryForm `alert()` E-14).

### Reference-quality templates (what "good" looks like here)

`NewPositionDialog` (zod + enforced required + `submitting` guard + correct colon-form i18n + 409-aware
retry + `applies_to` link matching the live CHECK), `PersonCreatePage` (zod, `disabled={isPending}`,
FormLabels, `dir="rtl"`), and `commitments.service.ts` (correct `aa_commitments` columns) are the
patterns the broken forms should be migrated toward.
