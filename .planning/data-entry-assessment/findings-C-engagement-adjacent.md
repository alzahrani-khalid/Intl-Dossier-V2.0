# Findings — Slice C: Engagement-adjacent data entry

**Scope:** positions (+ position↔dossier attach), after-action records (+ commitment extraction),
engagement briefs, MoUs, relationships, calendar entries.

**Summary.** The single most serious defect is the after-action **commitment insert**, which omits the
`NOT NULL` `aa_commitments.title` column — against the live schema every after-action saved _with a
commitment_ fails (HTTP 500) and rolls back the parent record (this is the real bug, **not** a "missing
table"). The calendar create flow accepts `participants` and `reminder_minutes` and **silently discards
both** while showing success. MoU creation is **not wired at all** (the "Add MoU" button and the
workflow-transition button are dead no-ops; no create form/route exists). Position attach/link error
branches swallow failures with `console.error` and no user feedback. Several MoU status labels and an
edit-mode recurring check are wrong. Brief writes, calendar table targeting, relationship writes, and
position `link_type` values were checked and are **correct** — and four loud agent CRITICALs were found
**false** on live-schema verification (see Rejected section).

All facts below are verified against the live staging schema (`zkrcjzdemdmwhearhfgg`) and the cited source.

**CRITICAL: 1 HIGH: 4 MEDIUM: 6 LOW: 8**

## Rejected on verification (do NOT chase — agents flagged these but live schema disproves them)

- **"`aa_commitments` table does not exist → every commitment 500s."** FALSE. The table **exists** (9 rows,
  full CHECK set). The edge fn writes to the _correct_ tables (`after_action_records`, `decisions`,
  `aa_commitments`, `aa_risks`, `aa_follow_up_actions`) and even satisfies the coupled `valid_owner` /
  `valid_tracking` CHECKs. The real bug is the omitted `NOT NULL` `title` → **C-1**.
- **"`ai_briefs` migration references a non-existent `ai_brief_status` enum → migration fails."** FALSE.
  Live DB has both the `ai_briefs` table and the `ai_brief_status` enum; the migration is applied.
- **"MoU `workflow_state` contract drift — `pending_approval`/`suspended`/`terminated` invalid."** FALSE.
  Those three **are** valid `mou_state` enum values (`draft,negotiation,pending_approval,signed,active,
suspended,expired,terminated`); current `MousPage.tsx:36-45` already matches the enum. The agent cited a
  **stale** `004_mous.sql` and stale code. (Only the _i18n labels_ for 3 of them are missing → C-8.)
- **"After-action `afterActions` i18n namespace unregistered → all AR strings break."** FALSE. The keys live
  under the default `common` namespace as a nested `afterActions` object in **both** `en/common.json` and
  `ar/common.json` (line ~571); `AfterActionForm` uses `useTranslation()` (default ns) so `t('afterActions.form.*')`
  resolves correctly in both languages.

## Verified-correct (checked, no defect — useful to record)

- **`engagement_briefs` is a non-insertable VIEW** (`is_insertable_into=NO`; `briefs UNION ALL ai_briefs WHERE
engagement_dossier_id IS NOT NULL`). Brief writes correctly go through edge fns into the **base** tables
  (`briefs` / `ai_briefs`) — the "direct view insert" trap is avoided. No fake-success found in the brief CTAs
  (mutations are awaited; toasts fire on real `onSuccess`/`onError`). (`briefs`=0/`ai_briefs`=0 rows in staging
  is a data-absence, not a code defect.)
- **Calendar create/update target `calendar_entries`** (the operational table), **not** the empty
  `calendar_events`. Uses the safe header-injected `getUser()` pattern (no `@2` 401 gotcha).
- **Positions `link_type` values are all valid** (`applies_to`/`related_to`/`endorsed_by`/`opposed_by`) at every
  call site; the frontend `PositionDossierLinkType` matches the DB CHECK exactly — no contract-drift on values.
- **Relationship create** (`/dossiers-relationships-create` → `dossier_relationships`) sends `status:'active'`
  (valid: active/historical/terminated) and an allowlisted `relationship_type`; write path is sound.
