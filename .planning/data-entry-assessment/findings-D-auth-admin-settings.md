# Findings — Slice D: Auth, admin & settings data entry

**Scope audited:** login/register/reset + MFA (enroll, step-up, login), profile/account settings,
AI org-LLM-policy settings, notification preferences, user-management/admin (create/role/clearance).

**Summary.** The recent edge-function security fixes landed correctly — the `123456` reset-password
backdoor is gone, the three MFA edge fns (`setup-mfa`/`verify-mfa-setup`/`reset-password`) now agree on
`public.users.mfa_secret` (base32, fail-closed), role reads are unified on `public.users.role`, and the
AI-settings payload matches the live `organization_llm_policies` columns (verified against staging). All my
slice's i18n namespaces (`settings`, `ai-admin`, `user-management`, `email-digest`) are registered EN+AR.
**But the data-entry layer underneath is broken or insecure in several load-bearing ways.** Two live RLS/grant
holes let any authenticated user self-elevate `role` and `clearance_level` by direct PostgREST write
(bypassing every server-side admin guard, which is the canonical authz source). The backend Express
`verifyMFACode()` is an always-true stub on live `/api/auth/*` routes, so MFA is bypassable there. The entire
Settings "Save" upserts ~24 columns that don't exist on `public.users` (every save fails), the live MFA
enrollment never writes a secret, the user-management create/role/clearance UI is dead-wired, and
`deactivate/reactivate-user` target a non-existent `status` column. Verification basis: live staging DB
`zkrcjzdemdmwhearhfgg` (RLS policies, table/column grants, triggers, row counts, storage buckets, edge-fn
deploy timestamps) + source reads; the highest-severity items were personally re-verified, not taken on an
agent's word.

**Counts:** CRITICAL: 3 HIGH: 11 MEDIUM: 17 LOW: 9

## Notes / positive verifications (the recent fixes that DID land)

- **Edge-fn MFA backdoor (N8) removed:** no `'123456'` / bypass literal in any deployed edge fn; `reset-password`
  does real TOTP (`otpauth`, ±1) against `public.users.mfa_secret`, fail-closed on missing secret. The
  surviving `123456` literal (D-34) is the NODE_ENV-gated contract mock only.
- **MFA secret persistence consistent:** `setup-mfa` writes `mfa_secret`(base32)+hashed `mfa_backup_codes`;
  `verify-mfa-setup` and `reset-password` read the same column/format. Column exists (`users.mfa_secret varchar`).
  (The live UI never reaches this path — D-6.)
