# v7.0 candidate — Intelligence Engine

## Premise

Product is "IntelDossier" but the codebase has a strong design shell and weak intelligence depth. v6.0 nailed the look. v7.0 should make the differentiator real: turn dossiers from passive records into a working intelligence layer.

## Current gaps that motivate this

- `intelligence_digest` table does not exist; digest source is `actor_name` (internal user) instead of a publication / origin signal — dashboard widget is a stub on top of a missing model
- "Signals" are referenced in CLAUDE.md and PROJECT.md but have no structured tracking, only ad-hoc commitments and after-actions
- Briefings are generated one-shot per dossier — no correlation across dossiers, no recurring digest, no alert thresholds
- Network graph is a static visualization (React Flow) without analytic queries on top — "who connects to whom across forums" is not a first-class query
- AI briefing uses Anthropic + OpenAI + local embeddings already, but only on demand from a single dossier context

## Possible scope for v7.0

| Theme                  | Outcome                                                                                                                                      |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| Intelligence model     | Schema for `intelligence_signal`, `intelligence_digest`, signal-to-dossier polymorphic linking; sources (publication / feed / human-entered) |
| Digest pipeline        | Recurring scheduled briefings per dossier or topic; cross-dossier digest aggregation; subscriber model (who sees which digest)               |
| Alerting               | Threshold-based alerts (new signal on tracked dossier, SLA breach correlation, VIP visit conflict); inbox + email + push                     |
| Analytic graph queries | "Who sits on which forum", "shared committees across organizations", "engagement chains over 30 days"; surface in Network panel and Cmd+K    |
| AI correlation         | Multi-dossier briefing (e.g. "all G20-adjacent activity this week"); pattern surfacing; prompt registry + audit trail                        |
| Data ingestion         | Scheduled feeds (RSS, public APIs, manual paste); deduplication; provenance tracking                                                         |

## Open questions to research before scoping

1. What signal types do diplomatic-ops users actually triage? (Domain probe — talk to a real analyst before building schema)
2. Is alerting an internal feature or does it need a vendor channel (Resend exists; what about SMS / Teams / Slack)?
3. Do we ingest external feeds in v7.0 or stay manual/AI-only?
4. Where does the Intelligence Engine sit relative to `Briefs` and `Operations Hub` in the IA — new top-level area or absorbed into Operations?
5. RBAC: which roles can create/subscribe/dismiss signals?

## Inputs

- `.planning/PROJECT.md` Validated Requirements list (line items reference Anthropic + OpenAI + local embeddings; pgvector; Network graph)
- `frontend/src/pages/Operations` — current Operations Hub already has 5 zones, candidate host for signal triage
- v6.0 deferred items that should land in v7.0 base layer: `intelligence_digest` table, VIP person ISO join (foreign-relations enrichment)

## Why this is the right v7.0 (vs other candidates)

- **vs Collaboration**: collaboration is horizontal infrastructure; valuable but doesn't differentiate against generic CRM/wiki competitors
- **vs Reporting/Export**: reporting is a downstream consumer of the intelligence model — building reports first means reporting on a shallow data layer
- **vs Workflow Depth**: lifecycle/MoU/after-action work is incremental on existing patterns; high effort / moderate novelty

The product name asks for this milestone. The design shell from v6.0 is ready to host it. The data layer needs to catch up.
