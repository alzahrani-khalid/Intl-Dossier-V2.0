**Gathered:** 2026-05-03

## Phase Boundary

Final QA gate for v6.0. Hard CI-level enforcement that the entire v6.0 surface area passes a quality bar across four axes:

1. **QA-01 — Logical-properties:** Zero `eslint-plugin-rtl-friendly/no-physical-properties` violations across every v6.0 page (Dashboard, Kanban, Calendar, all 7 list pages, Dossier drawer, Briefs, After-actions, Tasks, Activity, Settings — 13+ routes).
2. **QA-02 — A11y:** Zero axe-core WCAG 2A/2AA violations on every v6.0 route in EN/LTR + AR/RTL; keyboard-only navigation reaches every interactive target; focus outlines preserved across all 4 directions × both modes.
3. **QA-03 — Responsive:** Layout correct at 320 / 640 / 768 / 1024 / 1280 / 1536 px with ≥44×44px touch targets on every interactive element.
4. **QA-04 — Directional icons:** `arrow-right`, `arrow-up-right`, `chevron-right`, `chevron-left`, `.icon-flip`, and sparkline polylines flip via `scaleX(-1)` in RTL; `docs/rtl-icons.md` exists and enumerates every mirrored icon in the v6.0 set.

**In scope:**

- **Test infrastructure (Wave 0):**
  - Canonical v6.0 route registry at `frontend/tests/e2e/helpers/v6-routes.ts` exporting `V6_ROUTES: { name, path, requiresAuth, locales, hasMobileVariant }[]` — typed, single source of truth for cross-cutting sweeps.
  - Shared helpers at `frontend/tests/e2e/helpers/qa-sweep.ts`: axe runner (`runAxe(page, tags)`), breakpoint loop (`forEachBreakpoint(page, fn)`), focus-outline assertion (`assertFocusOutlineVisible`), Tab-walk helper (`tabWalkAllInteractives(page)`).
- **Cross-cutting sweep specs (Wave 1, parallel-safe):**

  - `frontend/tests/e2e/qa-sweep-responsive.spec.ts` — for each route × LTR + AR × {320, 640, 768, 1024, 1536}, runs render-assertion battery: (a) `document.body.scrollWidth ≤ viewport.width + 1`, (b) every visible interactive `boundingBox` has `width ≥ 44 && height ≥ 44`, (c) `<main>`, `<nav>`, page `<h1>`, primary CTA in viewport, (d) sidebar collapsed/hidden + mobile drawer trigger visible at ≤768px. 1280px keeps the existing per-phase visual baselines (Phase 38 D-13 / 40 D-06 / 41 D-12 / 42).

Catches focus-traps + hidden-but-tabbable + missing tabindex.
  - `frontend/tests/e2e/qa-sweep-focus-outline.spec.ts` — Settings page rendered at 4 directions × 2 modes = **8 visual baselines** with a focused element on each primitive type (button, input, link, toggle, nav row). **Out of scope (deferred):**

## Implementation Decisions

### Area 1: Test architecture

- **D-01: Hybrid — shared helpers + cross-cutting sweep specs, per-phase specs untouched.** Keep all 15+ existing per-phase a11y/rtl/visual/responsive specs as-is (already passing, well-tuned). Add (a) shared helpers in `frontend/tests/e2e/helpers/qa-sweep.ts`, (b) one cross-cutting `qa-sweep-axe.spec.ts`, (c) one `qa-sweep-keyboard.spec.ts`, (d) one `qa-sweep-responsive.spec.ts`, (e) one `qa-sweep-focus-outline.spec.ts`, (f) one opt-in `qa-sweep-icon-screenshots.spec.ts`.
  - **Why:** Failure isolation per phase stays intact; cross-cutting sweeps catch drift between phase teams. Aggregate-only would delete well-tuned specs and make a single failure hard to localize. Per-phase-extension would mean 13 PRs of churn + duplicated axe setup across teams.
- **D-02: Canonical route registry at `frontend/tests/e2e/helpers/v6-routes.ts`.** Typed TS module exporting `V6_ROUTES: { name, path, requiresAuth, locales, hasMobileVariant }[]`. Imported by all four cross-cutting sweep specs and any future sweep.
  - **Why:** Single source of truth, type-checked, easy to extend. Runtime derivation from `routeTree.gen.ts` is fragile to regen; JSON co-location loses type-checking.

### Area 2: Responsive sweep depth

- **D-03: Render-assertions only at 5 breakpoints (320/640/768/1024/1536); 1280px keeps existing per-phase visual baselines.** No new visual baselines added by Phase 43. Per-phase visual specs (`dashboard-visual`, `kanban-visual`, `calendar-visual`, `dossier-drawer-visual`, `list-pages-visual`, `briefs-page-visual`, `after-actions-page-visual`, `activity-page-visual`) stay at 1280 LTR+AR.
  - **Why:** Phase 38 D-13 / 40 D-06 / 41 D-12 / 42 precedent. 156 baselines (13 × 6 × 2) is unsustainable maintenance; render-assertion battery catches the actual regression class (overflow, broken touch targets, hidden landmarks) at a fraction of the cost.
- **D-04: Render-assertion battery — all four asserts run at every non-1280 breakpoint:**
  1. **No horizontal scroll:** `document.body.scrollWidth ≤ viewport.width + 1px tolerance`. Catches the most common responsive regression.
  2. **Touch targets ≥44×44px:** Loop visible `button, a, input, [role=button], [tabindex]:not([tabindex='-1'])`; assert each `boundingBox.width ≥ 44 && height ≥ 44`. Phase 40 D-18 / 41 D-11 / 42 D-18 precedent.
  3. **Key landmarks visible:** `<main>`, `<nav>`, page `<h1>`, primary CTA — boundingBox in viewport, not clipped, not `display:none`.
  4. **Sidebar / drawer state at ≤768px:** sidebar collapsed/hidden, mobile drawer trigger visible (Phase 36 shell-chrome contract).

### Area 3: rtl-icons.md scope & format

- **D-05: Comprehensive `docs/rtl-icons.md` with full audit table + LTR/RTL screenshots.** One markdown doc with three sections:
  1. **Flip mechanisms** — canonical patterns: `.icon-flip` class (CSS-only, used in `app.css` for table chevrons), inline `style={{ transform: 'scaleX(-1)' }}` (Phase 40 G3 row chevrons), locale-driven SVG transform (Phase 37 VIZ-05 Sparkline polyline), HeroUI `rotate-180` deprecation note.
  2. **Audit table** — every directional glyph in the v6.0 surface: icon name, source file, flip mechanism, RTL behavior, owning visual primitive. Researcher locates the full set; minimum baseline is the 5 named in QA-04 (`arrow-right`, `arrow-up-right`, `chevron-right`, `chevron-left`, `.icon-flip`) + sparkline polyline.
  3. **LTR/RTL screenshots** per documented icon, embedded as `![…](docs/rtl-icons/{icon}-ltr.png)` pairs.
- **D-06: Screenshots generated by an opt-in Playwright spec, committed under `docs/rtl-icons/`.** New `frontend/tests/e2e/qa-sweep-icon-screenshots.spec.ts` navigates a fixture page rendering each documented icon at fixed size; captures `{icon}-ltr.png` + `{icon}-rtl.png`. Run via `pnpm docs:rtl-icons` (not part of CI gate); screenshots regenerable when tokens change.
  - **Why:** Manual screenshots drift; inline SVG snippets miss the composed surface (icon inside a row with text). Generated + versioned screenshots stay current and reviewable as PR diffs.

### Area 4: Direction × mode × locale matrix

- **D-07: axe-core scope = 13 routes × LTR + AR @ Bureau-light = 26 sweeps.** Tags: `['wcag2a', 'wcag2aa', 'wcag21aa']`. Direction/mode change tokens (color, border, shadow), not roles/names/labels — semantic WCAG checks don't multiply across the 8-combo design matrix.
  - **Why:** 208-sweep full matrix is heavy CI cost with mostly duplicate findings. Color-contrast across the design matrix is covered by the focus-outline spec (D-08) + per-page Phase 38–42 visual baselines (which already render at Bureau-light by default).
- **D-08: Focus-outline visual baseline matrix = 1 route (Settings) × 4 directions × 2 modes = 8 baselines.** Settings has the most diverse interactive primitives in v6.0 (buttons, inputs, links, toggles, nav rows, switches). Render at each direction×mode combo with a focused element on each primitive type. Pair with programmatic assertion: enumerate combos via DesignProvider hatch, read `getComputedStyle(:focus-visible).outlineColor`, assert non-transparent + ≥3:1 contrast against background.
  - **Why:** Focus-ring is a global design-token property — per-route variation is rare. 1 representative route catches the matrix at 8 baselines instead of 104.
- **D-09: Keyboard navigation = per-route Tab-walk assertion in `qa-sweep-keyboard.spec.ts`.** For each route in `V6_ROUTES`: count visible interactives, press `Tab` N+1 times, assert (a) focused element changes each press, (b) set of focused elements equals the counted set exactly once (membership, not order). Catches focus-traps, hidden-but-tabbable, missing tabindex, programmatic `focus()` regressions.
  - **Why:** QA-02 mandates "reaches every interactive target" — only a Tab-walk satisfies that automatically. Tab-order golden diff over-constrains (legitimate order changes cause churn); axe-only doesn't trace the path; manual UAT drifts.

### Area 5: CI gate enforcement

- **D-10: All four sweeps are hard CI failures (block PR merges):**
  - `pnpm lint` fails on any `rtl-friendly/no-physical-properties` violation (already error-level; gate confirms)
  - `qa-sweep-axe.spec.ts` fails on any axe violation
  - `qa-sweep-responsive.spec.ts` fails on any render-assertion miss
  - `qa-sweep-keyboard.spec.ts` fails on any unreached interactive
  - **Why:** Phase 43 IS the gate — without blocking, regressions slip in and the milestone never converges. Focus-outline visual baselines and icon-screenshot specs run advisory (not blocking) so token tweaks don't ship-block on cosmetic drift.

### Claude's Discretion

- **Wave structure (3 waves, Phase 38/40/41/42 precedent):**
  - **Wave 0 (infra):** `helpers/v6-routes.ts` registry + `helpers/qa-sweep.ts` shared utilities + Playwright config tweaks (per-spec timeouts, retry policy) + `pnpm docs:rtl-icons` script + ESLint glob expansion.
  - **Wave 1 (parallel sweeps):** five sweep specs in parallel (`qa-sweep-axe`, `qa-sweep-responsive`, `qa-sweep-keyboard`, `qa-sweep-focus-outline`, `qa-sweep-icon-screenshots`) + `docs/rtl-icons.md` authored. Independent files; no merge conflicts.
  - **Wave 2 (gate + remediation):** wire CI gates; run all four blocking sweeps; fix every survivor inside Phase 43; commit baselines for focus-outline + icon screenshots.
- **Test runtime budget:** target full sweep <10 min on CI. If `qa-sweep-axe` × 26 sweeps blows the budget, shard by locale (LTR + AR projects) — researcher confirms current Playwright shard config.
- **Mock data vs real data:** sweeps run against the existing dev server / Playwright auth helper (`list-pages-auth.ts` pattern). No new fixture infra unless a route currently lacks deterministic content (researcher flags any route that 404s without auth context).
- **Tolerance for axe rules:** zero violations on `wcag2a/wcag2aa/wcag21aa` is the gate. `best-practice` tag is informational only (logged to PR comment, not blocking) — frequent false positives.
- **Direction probe in focus-outline spec:** use the existing `window.__design` test hatch from Phase 33-09 (`b01b5cd0`) to set direction/mode without UI clicking. Researcher confirms hatch still exposes `setDirection` and `setMode`.
- **Tab-walk filter:** exclude `<input type="hidden">`, `[aria-hidden="true"]`, and elements with `display:none` or `visibility:hidden` from the "visible interactives" count. Researcher confirms whether HeroUI Tabs/Modal portal targets are tabbable from the route they're triggered on (likely yes — they belong in the count).
- **Touch-target tolerance:** `≥44×44px` is the gate. `padding`-driven hit area (e.g., `<button className="p-3"><Icon size={16}/></button>`) counts if `boundingBox` ≥ 44; rejection mode is the actual rendered box, not the SVG glyph size.
- **Bilingual digit rendering** in any new test fixture pages → `toArDigits` (Phase 39/40/41/42 precedent).
