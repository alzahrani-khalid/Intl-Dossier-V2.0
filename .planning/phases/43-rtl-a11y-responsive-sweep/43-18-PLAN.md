Close survivor Gap-6 from 43-HUMAN-UAT post-execution: qa-sweep-keyboard
reports 26 routes with `reached.size < visibleCount` (out of 30 total
route × locale tests). 43-15's spec rebuild was correct — these are
genuine Tab-order failures, not test scaffolding bugs. The truthful
sweep now exposes them.

Two failure classes are likely (to be confirmed by Task 1 research):

1. **Spec over-counts**: `visibleCount` includes elements that look
   tabbable (`button`, `a[href]`, etc.) but actually aren't — they have
   `tabindex="-1"`, an `inert` ancestor, an `aria-hidden="true"`
   ancestor, or are display:contents wrapping a non-focusable. The
   visibility heuristic (offsetParent + rect) doesn't match Tab-order
   semantics.

2. **Production focus traps / unreachable interactives**: an element
   that SHOULD be in the Tab order has `tabindex="-1"` by mistake, or
   sits inside an `inert`/`aria-hidden` subtree that should be exposed,
   or its container has `pointer-events: none` blocking it.

Both classes need handling. Research first; fix second; do NOT skip the
research step or this plan ships papered-over failures.

Out of scope: route-specific UI redesign, modal/dialog focus management
(those have dedicated specs), shadow-DOM portal interactives.

### Task 2 — Spec refinement: visibleCount matches Tab-order semantics

Edit `frontend/tests/e2e/qa-sweep-keyboard.spec.ts`. Replace the
in-browser visibility filter inside `evaluateAll` with a Tab-order-aware
predicate:

```ts
const isTabbable = (el: HTMLElement): boolean => {
  // ancestor walk: any inert / aria-hidden / display:none / visibility:hidden
  // anywhere up the chain disqualifies the element
  let cur: HTMLElement | null = el
  while (cur && cur !== document.body) {
    if (cur.hasAttribute('inert')) return false
    if (cur.getAttribute('aria-hidden') === 'true') return false
    const cs = getComputedStyle(cur)
    if (cs.display === 'none' || cs.visibility === 'hidden') return false
    cur = cur.parentElement
  }
  // self: must have non-negative effective tabindex
  const ti = el.tabIndex
  if (ti < 0) return false
  // self: must have a real interactive box
  if (el.offsetParent === null) return false
  const rect = el.getBoundingClientRect()
  if (rect.width === 0 || rect.height === 0) return false
  return true
}
```

Apply `isTabbable(el)` instead of the current
`offsetParent !== null && rect non-zero` filter. Cluster-A/B/C/D/E
unreached cases are then correctly excluded from `visibleCount`.

Commit: `fix(43-18): align qa-sweep-keyboard visibleCount with Tab-order
semantics`.
