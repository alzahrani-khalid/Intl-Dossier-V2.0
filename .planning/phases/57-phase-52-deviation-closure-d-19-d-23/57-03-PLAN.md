<objective>
Close D-22 (LTR/RTL visual baseline byte-identity) by ensuring the i18next URL-param detector fires before React mounts → first render is in the target language → EN vs AR baselines diverge byte-wise. Add a hash-comparison check that mechanically enforces byte-distinction in CI, and add an ESLint guard preventing re-introduction of the addInitScript(i18nextLng) anti-pattern that caused the original byte-identity defect.

Purpose: Phase 52 D-22 carried PASS-WITH-DEVIATION because page.addInitScript writing localStorage.i18nextLng did NOT flip the language before first paint — i18next's localStorage detector reads `id.locale` (per frontend/src/i18n/index.ts:462), not `i18nextLng`. The 4 PNG baselines are therefore byte-identical between LTR and RTL. This plan fixes the spec-side language-flip mechanism by switching to URL params AND fixes the i18n config to include `querystring` in the detector chain.
