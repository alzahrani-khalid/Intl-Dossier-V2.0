**2. Implement `handleLifecycle` function:**

`GET /engagement-dossiers/:id/lifecycle` — Returns transition history:

- Query `lifecycle_transitions` WHERE `engagement_id = id` ORDER BY `transitioned_at DESC`.
- Left join `auth.users` or a profiles table to get `user_name` for display.
- Return `{ data: LifecycleTransition[] }`.

`POST /engagement-dossiers/:id/lifecycle` — Perform stage transition (LIFE-02, LIFE-03):

- Parse body: `{ to_stage: string, note?: string }`.
- Validate `to_stage` is one of the 6 valid stages. Return 400 if invalid.
- Do NOT validate direction — any stage to any stage is allowed per D-02, D-03.
- Read current `lifecycle_stage` from `engagement_dossiers` WHERE `id`.
- If `to_stage === current_stage`, return 400 "Already at this stage".
- In a single transaction (use Supabase RPC or sequential operations):
  a. INSERT into `lifecycle_transitions` with `from_stage = current_stage`, `to_stage`, `user_id = user.id`, `note` (null if not provided). The duration trigger handles `duration_in_stage_seconds`.
  b. UPDATE `engagement_dossiers` SET `lifecycle_stage = to_stage` WHERE `id`.
- Return `{ data: { lifecycle_stage: to_stage, transition_id: newId } }`.

**3. Implement `handlePromoteIntake` function** (LIFE-04, per D-06, D-07, D-08):

`POST /engagement-dossiers/promote-intake`:

- Parse body matching `IntakePromotionRequest`: `{ ticket_id, title_en, title_ar, objectives_en, objectives_ar, engagement_type, engagement_category, dossier_links }`.
- Validate ticket exists and has `request_type = 'engagement'` and `status !== 'converted'` per D-08. Return 400 if not promotable.
- Create a new dossier (base record) with `type: 'engagement'`, `name_en: title_en`, `name_ar: title_ar`.
- Create engagement extension in `engagement_dossiers` with:
  - `engagement_type`, `engagement_category` from request
  - `objectives_en`, `objectives_ar` from request
  - `lifecycle_stage: 'intake'` (per D-06)
  - `engagement_status: 'planned'` (default)
- If `dossier_links` provided, create entries in the appropriate linking table.
- Update the intake ticket: `status = 'converted'`, `convertedToType = 'engagement'`, `convertedToId = new_engagement_id`.
- Return `{ data: { engagement_id, dossier_id } }`.
- If extension insert fails after dossier creation, delete the base dossier to prevent orphans (per Research pitfall 6).

**4. Forum session creation** uses the existing `createEngagement` flow:

- However, verify the existing create handler passes through `parent_forum_id` to the extension insert. If it doesn't (likely — new column), add `parent_forum_id` to the extension insert logic in the existing create handler.
