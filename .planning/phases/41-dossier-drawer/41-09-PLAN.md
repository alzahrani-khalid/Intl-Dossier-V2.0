<objective>
Close VERIFICATION.md G3 + G4 — the axe-core color-contrast violation `text #fff9f8 on bg #bf5542 = 4.38 (need 4.5)` flagged by `dossier-drawer-axe.spec.ts` in BOTH locales (EN G3, AR G4). The failing token is `--sla-bad` (light mode), defined as `oklch(54% 0.20 25)` which renders as the hex `#bf5542`. White-ish ink (#fff9f8) on this bg falls 0.12 short of WCAG AA 4.5:1.

The fix is design-system-level, not per-component. Per CLAUDE.md "Visual Design Source of Truth" rules, color tokens are the single point of definition; consumers reference `var(--sla-bad)` and inherit the fix automatically. Plan 40-15 (commit `19c80358`) set the precedent — it darkened `--ok / --warn / --info` foregrounds for the same root cause (chip-warn / chip-ok / chip-info text on -soft backgrounds). `--sla-bad` was missed at that pass because no spec exercised its text-contrast surface; Phase 41's drawer axe gate is the first surface to scan a `.overdue-sev.high` (or sibling) chip with text.

**Wave assignment:** This plan runs in Wave 1 after Plan 41-08 (Wave 0). The Task 2 axe-core checkpoint requires the AR locale wait fix from Plan 41-08 Task 3 (G6) so that `dossier-drawer-axe.spec.ts` AR test can deterministically reach the rendered drawer for scanning. Without 41-08 Task 3, the AR axe check would race the i18n bootstrap and produce inconsistent results.

**Candidate replacement value: `oklch(46% 0.18 25)`.**
Rationale:

- Lightness 54% → 46% follows the same darkening delta Plan 40-15 used (~8 points).
- Chroma 0.20 → 0.18 mirrors the relationship between `--danger` (which is at 52% / 0.18) and `--sla-bad`. Reducing chroma slightly preserves perceptual saturation under the lower lightness.
- Hue 25 unchanged (the rust/coral red is deliberate; the handoff brand language uses it for severity).
- Resolved approximate hex: `#a14534` (oklch → sRGB conversion is approximate; executor MUST verify with axe-core that contrast against `#fff9f8` is ≥ 4.5:1 before committing).
- Symmetric: keeps the 8-point lightness delta between `--sla-bad-soft` (95% / 0.05 / 25) and `--sla-bad` exactly as the design system intends.

The executor MUST run an axe-core verification on the dashboard + drawer surfaces after editing the tokens — not just trust the math.

Purpose: Restore a WCAG AA-compliant `--sla-bad` token across every consumer (drawer, dashboard SLA strip, OverdueCommitments severity dots, and any future consumer) without per-component overrides.

Output: 4 token-file edits (1 production CSS, 1 buildTokens.ts, 2 handoff design-system files for byte-sync compliance with the FOUC bootstrap rule), one unit-test value adjustment in OverdueCommitments.test.tsx if the test asserts the literal hex (it does not — it asserts `var(--sla-bad)`, so no test changes are required). One axe-core verification step on the live dev server before commit.
