    2. i18n initializes asynchronously, reads `i18nextLng=ar` from localStorage, and fires `i18n.on('languageChanged', ...)` (line 486).
    3. The handler sets `document.documentElement.dir = 'rtl'`.

    Race: the test code that follows `loginForListPages` may execute before step 3 lands. The fix is for the helper to actively wait for the `dir` attribute to be `'rtl'` AFTER the reload.

    ```ts

      // Pre-seed localStorage so the language detector picks up AR on the next load.
      await page.evaluate(() => {
        window.localStorage.setItem('i18nextLng', 'ar')
        // Optimistic same-frame assertion in case anything reads dir before
        // i18n fires languageChanged. Will be re-asserted by the i18n bootstrap
        // on the next render.
        document.documentElement.setAttribute('dir', 'rtl')
        document.documentElement.setAttribute('lang', 'ar')
      })
      await page.reload()

      // Authoritative wait: i18n's languageChanged handler at frontend/src/i18n/index.ts:486-488
      // sets document.documentElement.dir = 'rtl' AFTER the languageChanged event fires.
      // This wait blocks until the live DOM matches the expected locale, so downstream
      // tests don't race the i18n bootstrap.
      await page.waitForFunction(
        () =>
          document.documentElement.getAttribute('dir') === 'rtl' &&
          document.documentElement.getAttribute('lang') === 'ar',
        null,
        { timeout: 10_000 },
      )
    }
    ```

    Then, AFTER the AR branch, the existing localStorage onboarding-dismissal block (lines 54-63) still runs unchanged.

    **Why a wait, not a refresh-loop:** The earlier helper relied on synchronous setAttribute then reload — but reload wipes the synchronous changes. Setting localStorage BEFORE reload is the deterministic approach (the language detector reads it on the next load), and `page.waitForFunction` is the correct Playwright primitive to bridge async DOM hydration.

    **Why this is cross-suite-safe:** The Phase 40 list-pages specs that already use `loginForListPages(page, 'ar')` will see the same authoritative wait — they will be slightly slower per test (≤ 200ms in practice — i18n loads quickly) but stricter. The wait timeout (10s) is large enough that no real test should hit it; if it does, the failure mode is loud (a clear timeout message pointing at the helper, not a flaky downstream assertion).
