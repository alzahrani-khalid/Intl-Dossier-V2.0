**Per D-11: Fix ONLY verified re-render issues. Do NOT add React.memo everywhere.**

**Step 1: Analyze widget-dashboard.tsx for re-render patterns.**

Read the full component tree in `widget-dashboard.tsx`. Look for:

- Inline object/function props passed to child components (these defeat memoization)
- Callback functions defined inside render that should be hoisted with useCallback
- Computed values recalculated every render that should use useMemo
- Child components that receive stable data but re-render due to parent state change

Common patterns to fix:

1. If widget cards receive `style={{ ... }}` inline objects, extract to stable references via useMemo
2. If the dashboard renders a grid of widgets and each widget gets an inline `onAction={() => handleAction(widget.id)}`, wrap with useCallback or create a stable handler map
3. If dashboard maps over widget data and creates complex JSX for each, memoize the individual WidgetCard component with React.memo

Apply fixes:

- Wrap individual widget card components with `React.memo` if they receive stable props but re-render on parent state changes
- Use `useMemo` for derived data (filtered lists, sorted arrays, computed stats) that is passed as props
- Use `useCallback` for event handlers passed to child components
- Hoist constant objects (static configs, empty arrays, default values) OUTSIDE the component

**Step 2: Analyze kanban-board.tsx for re-render patterns.**

Read the full component. Kanban boards are classic re-render offenders because:

- Moving a card between columns triggers full board re-render
- DnD state changes cause all columns to re-render
- Each column renders all its cards, and unaffected columns re-render needlessly

Apply fixes:

- Memoize individual KanbanColumn component with `React.memo` so columns only re-render when their items change
- Memoize individual KanbanCard component with `React.memo` so cards only re-render when their data changes
- Use `useMemo` for per-column item filtering: `const columnItems = useMemo(() => items.filter(i => i.status === column.id), [items, column.id])`
- Use `useCallback` for DnD handlers passed to columns/cards
- If DnD overlay/drag state lives in parent, ensure it does NOT pass through to non-dragged columns

**Important constraints (per D-11 anti-pattern warnings):**

- Do NOT add React.memo to components that receive inline objects without also stabilizing the objects
- Do NOT memoize components that are cheap to render (simple text, icons)
- DO add comments explaining why each memo/useMemo/useCallback was added: `// Memo: prevents re-render of column when unrelated column's items change`

Read the full component. The dossier list is a high-traffic page that renders many cards/rows. Look for:

- List items that re-render when any item in the list changes
- Filter/sort state changes causing full list re-render
- Search input re-rendering the entire list on each keystroke

Apply fixes:

- Memoize individual dossier card/row components with `React.memo` if they exist as separate components in the file

- Tabs that all re-render when the active tab changes
- Sidebar info that re-renders when main content changes
- Related entity lists that re-render on any detail change

Apply fixes:

- Memoize tab content components with `React.memo` so inactive tabs don't re-render
- Use `useMemo` for derived data (relationship counts, status badges, computed fields)
- Use `useCallback` for handlers passed to child components

**Step 3: Audit context providers for splitting (per D-12).**

Read `AuthContext.tsx`, `theme-provider.tsx`, `language-provider.tsx`.

For each context, determine:

1. How many values does it provide?
2. How frequently does each value change?
3. Do consumers use ALL values or just specific ones?

**AuthContext analysis:** If AuthContext provides `{ user, session, role, permissions, loading, error }` and some components only need `user.id` while others need `role`, consider splitting into:

- `AuthStateContext` (stable after login: user, role, permissions)
- `AuthSessionContext` (changes more frequently: session, loading, error)

**However per D-12:** Only split if you find evidence that cascading re-renders are a real problem. Evidence patterns:

- Many components import useAuth() but only read one field
- Loading state changes trigger re-renders in components that only need user data

If the context is simple (1-3 values, stable after login), do NOT split -- document finding as "Context splitting not needed: {reason}".

If splitting IS warranted, apply the split:

```typescript
// Before: one context with everything
const AuthContext = createContext<AuthContextType>(...)

// After: split into stable + dynamic
const AuthIdentityContext = createContext<{ user: User, role: string }>(...) // Stable after login
const AuthSessionContext = createContext<{ session: Session, loading: boolean }>(...) // Dynamic
```

For ThemeProvider and LanguageProvider: These change very rarely (only on user toggle), so splitting is almost certainly NOT needed. Document this finding.
