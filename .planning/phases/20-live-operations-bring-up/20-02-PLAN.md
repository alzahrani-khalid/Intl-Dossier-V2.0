## Runbook Steps

### Step 1 — Create A record at registrar

**Action:** In the DNS control panel of the registrar, create an A record:

- Host: `@` (or the chosen subdomain, e.g. `intl-dossier`)
- Type: `A`
- Value: `138.197.195.242`
- TTL: 300 (low, so you can iterate)

**Expected result:** DNS entry saved.

**Verification (local shell):**

```bash
dig +short A intl-dossier.gastat.gov.sa
# expect: 138.197.195.242
```

**Human checkpoint:** `dig` returns the droplet IP. Save output to `evidence/20-02/01-dig-a-record.txt`. Wait until propagation is stable before Step 2 (retry `dig` every 2 minutes if needed — do NOT run certbot prematurely).

### Step 2 — Apply domain to nginx config (replace `_DOMAIN_` placeholder)

**Action:** SSH to droplet and replace the `_DOMAIN_` placeholder (per `14-01-PLAN.md` Fix 1) in `nginx.prod.conf`. Before running `sed`, guard against a no-op re-run:

```bash
grep _DOMAIN_ deploy/nginx/nginx.prod.conf || echo "already substituted — skip sed"
```

Then:

```bash
ssh root@138.197.195.242 "cd /opt/intl-dossier && git pull && \
  sed -i 's/_DOMAIN_/intl-dossier.gastat.gov.sa/g' deploy/nginx/nginx.prod.conf && \
  cd deploy && docker compose -f docker-compose.prod.yml up -d nginx"
```

**Expected result:** nginx reloads with real domain in cert paths.

**Verification:**

```bash
ssh root@138.197.195.242 "docker exec intl-dossier-nginx nginx -t"
```

**Human checkpoint:** `nginx -t` prints `syntax is ok` and `test is successful`.

### Step 3 — Run certbot to obtain the certificate

**Action:**

```bash
ssh root@138.197.195.242 "docker run --rm \
  -v /opt/intl-dossier/letsencrypt:/etc/letsencrypt \
  -v /opt/intl-dossier/certbot-www:/var/www/certbot \
  --network intl-dossier-prod \
  certbot/certbot certonly --webroot -w /var/www/certbot \
  -d intl-dossier.gastat.gov.sa \
  --email ops@gastat.gov.sa --agree-tos --non-interactive"
```

**Expected result:** `Successfully received certificate. Certificate is saved at: /etc/letsencrypt/live/intl-dossier.gastat.gov.sa/fullchain.pem`

**Verification:**

```bash
ssh root@138.197.195.242 "ls -la /opt/intl-dossier/letsencrypt/live/intl-dossier.gastat.gov.sa/"
```

**Human checkpoint:** `fullchain.pem` + `privkey.pem` present. Output saved to `evidence/20-02/02-certbot-success.txt`.

**IMPORTANT — Let's Encrypt rate limit**: if this step fails, **do not loop**. Fix the underlying issue (DNS not propagated, firewall blocking :80, webroot path wrong) then retry at most 2–3 times. You have 5 duplicate certs per week.

### Step 4 — Reload nginx to pick up the cert

**Action:**

```bash
ssh root@138.197.195.242 "cd /opt/intl-dossier/deploy && \
  docker compose -f docker-compose.prod.yml restart nginx"
```

**Expected result:** nginx serving HTTPS.

**Verification:**

```bash
curl -sI https://intl-dossier.gastat.gov.sa/health
```

Expect `HTTP/2 200` and a `strict-transport-security` header.

### Step 5 — Run `verify-deployment.sh` against the real domain

**Action (local shell, from repo root):**

```bash
DOMAIN=intl-dossier.gastat.gov.sa bash deploy/verify-deployment.sh
```

**Expected result:** All 5 checks PASS (health, API proxy, TLS certificate, HSTS, HTTP→HTTPS redirect).

**Verification:** Full stdout captured.

**Human checkpoint:** Exit code 0, all checks PASS. Output saved to `evidence/20-02/03-verify-deployment.txt`. — Closes **success criterion 5** (DEPLOY-01).

### Step 6 — Generate a dedicated deploy SSH key pair

**Action (local shell):**

```bash
ssh-keygen -t ed25519 -C "gh-actions-deploy" -f ~/.ssh/intl_dossier_deploy -N ""
```

**Expected result:** Two new files — `~/.ssh/intl_dossier_deploy` (private) and `~/.ssh/intl_dossier_deploy.pub` (public).

### Step 7 — Authorize the public key on the droplet

**Action:**

```bash
ssh root@138.197.195.242 "mkdir -p ~/.ssh && chmod 700 ~/.ssh && \
  cat >> ~/.ssh/authorized_keys" < ~/.ssh/intl_dossier_deploy.pub
```

**Verification:**

```bash
ssh -i ~/.ssh/intl_dossier_deploy root@138.197.195.242 "echo ok"
# expect: ok
```

**Human checkpoint:** Test SSH using the new key succeeds.

### Step 8 — Store `DROPLET_HOST` and `DROPLET_SSH_KEY` as GitHub Actions secrets

**Action:** Repo → Settings → Secrets and variables → Actions → New repository secret:

- `DROPLET_HOST` = `138.197.195.242`
- `DROPLET_SSH_KEY` = full contents of `~/.ssh/intl_dossier_deploy` (private key, including BEGIN/END lines)
- `DROPLET_SSH_USER` = `root` (if workflow expects it)

**Expected result:** Secrets saved.

**Human checkpoint:** Three secrets visible in list. Screenshot → `evidence/20-02/04-gh-deploy-secrets.png` (redact values).

## Rollback

- **If DNS change breaks unrelated subdomains:** delete the A record; fallback is to use a throwaway staging subdomain.
- **If certbot hits rate limit:** stop retrying. Wait for the rate-limit window to expire OR use `--staging` flag to iterate (staging certs are untrusted but unlimited), then re-run without `--staging` once you're sure the config works.
- **If the first live deploy fails after image swap:** `deploy.yml` has automatic rollback per `14-VERIFICATION.md` row DEPLOY-05. Otherwise run manually:
  ```bash
  ssh root@138.197.195.242 "cd /opt/intl-dossier/deploy && bash rollback.sh"
  ```
- **If SSH key leaks:** immediately remove from droplet `authorized_keys` and rotate the GH secret.
