The risk is amplified because the function is SECURITY DEFINER: it runs as owner and BYPASSES the `dossiers` SELECT RLS policy `sensitivity_level <= caller clearance`. I verified live that `engagement_participants`/`engagement_dossiers`/`dossiers` all have RLS enabled with permissive, clearance/archived-gated SELECT policies â€” but those policies are NOT evaluated inside a DEFINER function. So the new `recent_engagements` branch returns engagement `name_en`/`name_ar`/`location_en`/`location_ar` for engagements ABOVE the caller's clearance, and granting `anon` makes that reachable by the unauthenticated PostgREST role. (Note: I confirmed the LIVE function currently already has an anon grant â€” likely from a prior ad-hoc grant or PostgREST default â€” so on staging this is status-quo, but the repo migration is now codifying anon access that `20260202000001` never gave. In practice the `persons` edge fn is auth-walled, which mitigates real-world exposure, but the SQL grant itself is broader than the prior contract and broader than necessary.)

**Adversarial verification:** Verified against the actual files and live DB. (1) Grant widening is real: migration L88 grants `get_person_full(uuid)` TO `authenticated, anon, service_role`, whereas the only prior repo grant â€” `20260202000001` L687 â€” was `GRANT EXECUTE ON FUNCTION get_person_full TO authenticated;` (authenticated only). **Suggested fix:** Drop `anon` (and `service_role` unless deliberately needed) from the GRANT to match the original contract: `GRANT EXECUTE ON FUNCTION public.get_person_full(uuid) TO authenticated;`. The 'info' severity is appropriate: a genuine hardening opportunity on a security-sensitive DEFINER+anon surface, but pre-existing and not newly exploitable beyond what already shipped. Not stylistic noise.

**Suggested fix:** Add a one-line `ALTER FUNCTION public.get_person_full_base(uuid) SET search_path = public, pg_temp;` right after the RENAME (inside or after the DO block) to pin the base the same way the wrapper is pinned. Cheap, idempotent, closes the pre-existing DEFINER search_path gap.
