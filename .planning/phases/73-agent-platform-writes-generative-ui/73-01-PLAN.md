<objective>
Lay the two DB foundations P73 writes require so the frontend approve-handlers (73-03) can commit under the caller's JWT and the actor can be asserted (GENUI-03):

1. **Signal actor migration (D-06):** `intelligence_event` records `status` but NOT who changed it. Add `acknowledged_by`, `dismissed_by`, `escalated_by` (uuid → `auth.users`) and `status_changed_at` so an approved dismiss/escalate write writes a verifiable `actor_id == auth.uid()`.
2. **`persist_brief` INVOKER RPC (D-02), RECONCILED to the LIVE briefs schema:** The two existing brief-write paths are both disqualified — the AI path (`dossiers-briefs-generate`) calls an external LLM (P74 retires it) and the manual path (`engagement-briefs`) switches to the service-role client (violates GENUI-03). Add a NEW `SECURITY INVOKER` `persist_brief(p_dossier_id, p_content, p_title, p_summary)` RPC that inserts into the **LIVE** `public.briefs` table (the rich jsonb-content schema — `type/target_entity/purpose/audience/parameters/content/generation/version/tenant_id/organization_id/is_deleted/status`, NOT the dead `content_en/content_ar/dossier_id`) under the caller JWT, pinning `created_by = auth.uid()`. The bilingual draft is carried in the single `content` jsonb as `{ en:{summary,sections}, ar:{summary,sections} }`. Because the live briefs RLS keys org-isolation on the **empty** `org_id` JWT claim AND `dossiers` carries no org, the RPC resolves `organization_id == tenant_id` via `tenant_isolation.get_current_tenant_id()` (the exact source the proven INVOKER `publish_digest` uses). Authorization is an **additive** `briefs_insert_via_dossier_edit` INSERT policy gating on a new additive `source_dossier_id` link column via `can_edit_dossier()`; the RPC returns NULL on a non-editable dossier (indistinguishable-empty). NO content-column migration is performed.

Purpose: Both are pure DB prerequisites with no agent/frontend coupling — they unblock every later wave.
Output: Two idempotent migrations applied to staging via the Supabase MCP.
</objective>
