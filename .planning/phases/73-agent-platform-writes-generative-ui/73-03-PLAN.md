makeAssistantToolUI<TArgs, TResult>({ toolName, render }) → component; mount the returned components
as children under AssistantRuntimeProvider (they register on mount). `useAssistantToolUI({ toolName, render })`
is the hook-equivalent if a component lifecycle is awkward.

render props (0.14.x tool-call mechanism): { args, status, result, addResult, toolCallId }

- args = the propose\_\* tool's echoed args (see 73-02 SUMMARY for exact schemas)
- addResult({ ... }) = the HITL roundtrip: call it on Approve/Reject to resolve the tool call.
  Approve → addResult({ approved: true, committed: <true|false>, ... }); Reject → addResult({ approved: false, reason }).
- render post-state from `status`/`result`.
  DISTINCTION (73-RESEARCH §B): CitationCard uses the SOURCE message-part mechanism; HITL cards use the
  TOOL-CALL mechanism (makeAssistantToolUI). This plan introduces the first tool-UI renderers.

propose\_\* tool ids + args the cards key to (from 73-02):
propose_publish_digest → { dossierId, period: daily|weekly|monthly, summary }
propose_signal_status → { signalId, action: dismiss|escalate, reason? }
propose_work_item → { title, assigneeId, priority?, dossierIds[], inheritanceSource? }
propose_brief → { dossierId, content: { en:{summary,sections}, ar:{summary,sections} } }

The authenticated supabase client (caller JWT) is `import { supabase } from '@/lib/supabase'` (same client
useSignalMutations uses). NEVER import or construct a service-role client in the frontend.

Existing INVOKER commit targets (commit under the caller JWT):

- digest: supabase.rpc('publish_digest', { p_dossier_id, p_period, p_summary, p_clearance_level_at_generation })
  (clearance level = the caller's profiles.clearance_level; read it the way publish_digest expects, or pass the value the agent had — confirm against the RPC body in 73-01's read_first.)
- signal: supabase.from('intelligence_event').update({ status, <dismissed_by|escalated_by>: uid, status_changed_at: now }).eq('id', signalId)
  (dismiss→status 'dismissed' + dismissed_by; escalate→status 'escalated' + escalated_by. The D-06 columns exist after 73-01.)
- work item: POST tasks-create edge fn ({ title, assignee_id, priority, ... }) → then POST work-item-dossiers ({ work_item_type:'task', work_item_id, dossier_ids, inheritance_source }). Both forward the caller Bearer.
- brief: supabase.rpc('persist_brief', { p_dossier_id, p_content, p_title, p_summary }) — RECONCILED to the live briefs schema. `p_content` is the SINGLE jsonb envelope from propose_brief's `content` ({ en:{summary,sections}, ar:{summary,sections} }); `p_title`/`p_summary` are optional (pass the EN summary's title/summary or null). There is NO p_content_en/p_content_ar/p_engagement_dossier_id arg — the live briefs table has no content_en/content_ar/dossier_id column and persist_brief resolves org/tenant itself. A non-editable dossier returns NULL (no row) — treat a NULL return as a neutral failure.

Query key factories (post-commit invalidation):
signalKeys.lists() (frontend/src/domains/signals/types/signal.types.ts)
workItemKeys.byDossier(dossierId), workItemKeys.lists() (frontend/src/domains/work-items/keys.ts)
dossierKeys.detail(dossierId) (frontend/src/domains/dossiers/hooks/useDossier.ts)
brief keys / digest keys: reuse the existing factory for those domains if present; otherwise invalidate dossierKeys.detail(dossierId) (briefs/digests render under the dossier).
