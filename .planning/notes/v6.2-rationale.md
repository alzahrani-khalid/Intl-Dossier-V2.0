# v6.2 Type-Check, Lint & Bundle Reset — Rationale

## Decision

Run a surgical milestone (v6.2) to restore code-quality gates and bundle budget before scoping v7.0 Intelligence Engine.

## Why before v7.0

Investigation on 2026-05-08 surfaced quality-gate failures invisible to the v6.0 / v6.1 audit machinery (audits scored requirements coverage and visual deviation, not pipeline health):

| Surface                    | Result on `main`                                                                       |
| -------------------------- | -------------------------------------------------------------------------------------- |
| `frontend pnpm type-check` | FAIL — 1580 TS errors (TS6133/TS6196: unused declarations and unused exported types)   |
| `frontend pnpm lint`       | FAIL — 52 errors + 671 warnings (723 problems total)                                   |
| `backend pnpm type-check`  | FAIL — 498 TS errors (same TS6133 shape)                                               |
| `backend pnpm lint`        | FAIL — 3 errors + 1 warning                                                            |
| `frontend pnpm size-limit` | Total JS = 2.42 MB gzipped vs aspirational 200 KB v2.0 target (47.4 s on slow-3G load) |

Implications:

- CI must be allowing type-check / lint to pass despite these errors, OR the pipeline does not run them on PRs. Either way, no static type or lint enforcement blocks bad code from `main` today.
- The v6.0 audit raised `size-limit` Total JS to 815 KB to absorb vendor super-chunk drift. Phase 44 repaired the chunk-glob gate. Today the ceiling is **2.43 MB** (raised again) — the gate is symbolic, not enforcing. v2.0 Phase 7's 200 KB budget is 12× exceeded.
- `frontend/eslint.config.js` still references Aceternity UI library (`3d-card`, `bento-grid`, `floating-navbar`, `link-preview`) in `no-restricted-imports` warnings, contradicting CLAUDE.md which bans Aceternity without explicit user request.

Building v7.0 (intelligence_signal schema, digest pipeline, alerting, multi-dossier AI correlation) on top of these failures means:

- New schema and TS types land without compile-time enforcement; type-shaped bugs reach staging.
- Pgvector and embeddings work tempts more dependencies → bundle worsens before it is fixed.
- Aceternity / shadcn drift in the lint config will compound as new components are added.

Cheap to fix as a batch. Expensive to fix later.

## Scope (3 phases, ~6-8 plans estimated)

| Phase | Title               | Source                                                                                                                                              | Plans est. |
| ----- | ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- | ---------- |
| 47    | Type-check zero     | 1580 frontend + 498 backend TS errors → 0; restore type-check CI gate; document deletion vs preservation policy for unused exports                  | 2-3        |
| 48    | Lint zero           | 52 frontend errors + 671 warnings → 0; 3 backend errors → 0; remove Aceternity references from `no-restricted-imports`; align config with CLAUDE.md | 2          |
| 49    | Bundle budget reset | Real target (proposal: ≤500 KB initial, route-split heavy chunks, vendor super-chunk audit); restore size-limit gate as PR-blocking                 | 2-3        |

## Risk

Low–medium. Most TS errors are unused declarations (mass-delete or `knip` auto-fix). Lint warnings 90% same class. Real risk: enabling strict checks may surface hidden type bugs in actively-used code paths — that is the point.

Bundle reset carries higher risk (route-split breaks lazy boundaries; vendor chunking changes hash names). Mitigate with focused E2E re-run.

## Deferred

- **Intelligence Engine** (signals model, digest pipeline, alerting, multi-dossier AI correlation) — captured as seed `v7.0-intelligence-engine.md`. Trigger updated: v6.2 ships.
- **Visual deviation closeouts** for Phase 38/39/41 PASS-WITH-DEVIATION items — separate concern, not blocking v7.0.
- **Storybook** — ADR-006 stands; existing Vitest + Playwright coverage is acceptable until primitive count > 15.

## Open question

`.planning/research/questions.md` Q1 — Are TS errors and lint failures auto-suppressed in CI today, or is type-check not in the deploy pipeline at all? Answer determines whether Phase 47 also needs a CI-gate plan.
