### Step 3 — Rehearse Supabase restore against STAGING

> **Source vs Target:** The _source_ backup is the current live Supabase project. The _target_ is a NEW throwaway Supabase project the operator creates for this rehearsal — NOT `zkrcjzdemdmwhearhfgg` (that is permanent staging and must not be overwritten). Delete the throwaway project after evidence is captured.

**WARNING:** This step operates on Supabase infrastructure. Double-check the target project ID before every command. `deploy/BACKUP_RESTORE.md` Known Risks:

- Role passwords NOT preserved (reset `postgres` password after restore)
- `pg_net` and `pg_cron` may fire unintended actions — disable immediately after restore
- Extension version conflicts may fail restore

**Action (per `deploy/BACKUP_RESTORE.md` Restore Procedure):**

1. In Supabase dashboard, go to `https://supabase.com/dashboard/project/zkrcjzdemdmwhearhfgg` → Database → Backups.
2. Select a recent backup date. Download the backup file.
3. Create (or re-use) a throwaway staging project. Record its project ID.
4. Restore into the staging project:
   ```bash
   psql "postgresql://postgres:[password]@db.[staging-project-id].supabase.co:5432/postgres" < backup.sql
   ```
5. Immediately disable `pg_net` and `pg_cron` on the restored project (SQL editor):
   ```sql
   drop extension if exists pg_net;
   drop extension if exists pg_cron;
   ```
6. Reset the `postgres` role password via staging project dashboard.
7. Data-consistency sanity checks:
   ```sql
   select count(*) from dossiers;
   select count(*) from work_items;
   select count(*) from public.users;
   select max(updated_at) from dossiers;
   ```

**Expected result:** Row counts approximately match production at backup time. No errors during restore.

**Human checkpoint:** Restore completed, counts recorded, pg_net/pg_cron disabled, password reset done. Write a short log to `evidence/20-03/03-restore-rehearsal-log.md` including: backup date, target project ID, before/after row counts, any warnings. — Closes **success criterion 9** (DEPLOY-04b).

## Rollback

- **If Redis backup script fails:** check file ownership, `/opt/intl-dossier/backups/redis` exists and is writable, `docker exec intl-dossier-redis redis-cli BGSAVE` is permitted.
- **If Supabase restore corrupts staging:** delete the staging project and re-create it from scratch. No production impact.
- **If `rollback.sh` leaves the droplet in a broken state:** run Step 6 (roll forward via GH Actions). As a last resort, SSH in and manually re-pull latest tags per `deploy/DROPLET_INSTRUCTIONS.md`:
  ```bash
  ssh root@138.197.195.242 "cd /opt/intl-dossier && git pull && cd deploy && \
    docker compose -f docker-compose.prod.yml build && \
    docker compose -f docker-compose.prod.yml up -d"
  ```
