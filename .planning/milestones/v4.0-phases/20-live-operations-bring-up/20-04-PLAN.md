# Phase 20 — Plan 04: Monitoring & Alerting

**Phase:** 20 — Live Operations Bring-Up
**Plan:** 20-04
**Title:** Monitoring & Alerting (UptimeRobot or Betterstack on /health, verified test alert)
**Goal:** Ensure we know within minutes when the site is down, by wiring an external uptime monitor at `https://intl-dossier.gastat.gov.sa/health` and confirming a test alert is delivered to on-call.
**Requirements:** DEPLOY-03
**Depends on:** Plan 20-02 (domain + TLS live)

## Context

`14-VERIFICATION.md` Human item 3 recorded DEPLOY-03 as `human_needed`: the `/health` endpoint exists and docs in `14-03-PLAN.md` Task 2 describe the setup, but no human has signed up for UptimeRobot/Betterstack and wired an alert channel.

**Gap closed:** `14-VERIFICATION.md` Human item 3 (DEPLOY-03).

## Prerequisites

- Plan 20-02 complete (public HTTPS endpoint exists)
- Operator email or phone for alert delivery
- An UptimeRobot OR Betterstack account (free tier is sufficient for one monitor)

## Runbook Steps

### Step 1 — Choose monitor provider and sign up

**Action:** Pick one:

- **UptimeRobot**: https://uptimerobot.com (free tier: 50 monitors, 5-min interval)
- **Betterstack**: https://betterstack.com/uptime (free tier: 10 monitors, 3-min interval)

Sign up using an ops-owned email (NOT a personal address).

**Human checkpoint:** Account created, email verified.

### Step 2 — Create a monitor pointing at `/health`

**Action:** In the provider dashboard, create a new HTTP(s) monitor:

- Name: `Intl-Dossier Production /health`
- URL: `https://intl-dossier.gastat.gov.sa/health`
- Method: `GET`
- Expected status: `200`
- Interval: 5 minutes (UptimeRobot free) or 3 minutes (Betterstack free)
- Region: Europe (closest to eu-west-2 droplet)
- Optional keyword check: `ok` or similar — only if `/health` returns a body containing a known token

**Expected result:** Monitor appears in the dashboard and within one interval reports status `UP`.

**Verification:** Dashboard shows green / UP.

**Human checkpoint:** Monitor UP. Screenshot → `evidence/20-04/01-monitor-up.png`.

### Step 3 — Configure alert contacts

**Action:** Add at least two alert channels:

- Email: operator's ops email
- (Optional but recommended) SMS or Slack webhook or PagerDuty integration

Link both contacts to the new monitor. Set thresholds:

- Alert after: 2 consecutive failures (avoids single-probe flap)
- Notify on: DOWN and UP (so we know when recovery happens)

**Human checkpoint:** At least two alert channels attached. Screenshot → `evidence/20-04/02-alert-contacts.png`.

### Step 4 — Fire a deliberate test alert

**Action:** Two options — do ONE:

**Option A (recommended — use provider's built-in "send test alert" feature):**
In UptimeRobot: Monitor → Actions → `Test alert contacts`. In Betterstack: Monitor → `Send test notification`.

**Option B (real induced outage, only if you control a maintenance window):**

```bash
ssh root@138.197.195.242 "cd /opt/intl-dossier/deploy && \
  docker compose -f docker-compose.prod.yml stop nginx"
# wait until monitor fires (up to 2 × interval)
ssh root@138.197.195.242 "cd /opt/intl-dossier/deploy && \
  docker compose -f docker-compose.prod.yml start nginx"
```

**Expected result:** Operator's email (and SMS/Slack if configured) receives a DOWN alert. A recovery alert follows when the site comes back.

**Human checkpoint:** Alert received in inbox. Screenshot of the alert email (or Slack message, or SMS) → `evidence/20-04/03-test-alert.png`. — Closes **success criterion 7** (DEPLOY-03).

### Step 5 — Document runbook for on-call

**Action:** Add a one-line entry to `deploy/DROPLET_INSTRUCTIONS.md` (only if missing) under an "On-Call" section documenting:

- Monitor provider + URL
- Who receives alerts
- Link to `deploy/rollback.sh` and `deploy/verify-deployment.sh`

This is a tiny doc edit, not new product code — acceptable within "bring-up" scope.

**Human checkpoint:** Doc edit committed (or skipped with explicit note if `DROPLET_INSTRUCTIONS.md` already covers it).

## Rollback

- **If test alert does not arrive:** verify contact email isn't in spam, verify provider dashboard shows an alert was sent, re-run Step 4 Option A. Do NOT leave Option B's `nginx` in a stopped state — always run the `start` command.
- **If the monitor is pointed at a wrong URL:** delete and recreate; no side effects.
- **Account deletion:** simply delete the monitor provider account if abandoned.
