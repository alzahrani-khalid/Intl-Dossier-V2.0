<threat_model>

## Trust Boundaries

| Boundary          | Description                                                                 |
| ----------------- | --------------------------------------------------------------------------- |
| Service → Queue   | Internal server enqueues jobs; no external input directly reaches the queue |
| Worker → Supabase | Worker uses service role key to insert notifications; bypasses RLS          |
| Redis connection  | BullMQ connects to Redis; must be same-host or private network              |

## STRIDE Threat Register

| Threat ID | Category               | Component                 | Disposition | Mitigation Plan                                                                                                                                                                                |
| --------- | ---------------------- | ------------------------- | ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| T-15-01   | Tampering              | notification.processor.ts | mitigate    | Validate NotificationJobData fields with TypeScript interface at enqueue time; worker rejects jobs missing required fields (userId, type, title, message, category)                            |
| T-15-02   | Info Disclosure        | queue-connection.ts       | mitigate    | Redis URL from env var only; never logged; connection uses same-host or private network                                                                                                        |
| T-15-03   | Denial of Service      | notificationQueue         | mitigate    | BullMQ concurrency: 5 limits parallel processing; removeOnComplete/removeOnFail prevents unbounded queue growth; attempts: 3 with exponential backoff prevents retry storms                    |
| T-15-04   | Elevation of Privilege | notification.processor.ts | accept      | Worker uses SUPABASE_SERVICE_ROLE_KEY (bypasses RLS) but only inserts into notifications table for the specified user_id; no user-controlled escalation path since enqueue is server-side only |

</threat_model>
