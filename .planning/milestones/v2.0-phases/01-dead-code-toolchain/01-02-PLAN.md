**Step 1: Run initial Knip scan to see all findings.**

```bash
pnpm exec knip --no-progress 2>&1
```

Categorize the output into: unused files, unused exports, unused dependencies, unlisted dependencies.

**Step 2: Handle unused dependencies (TOOL-04).**

For each unused dependency Knip reports:

- If the package is genuinely unused (no import chain): `pnpm remove <pkg>` from the appropriate workspace
- If Knip is wrong (dynamic import, runtime-only dep like `tsx`): add to `ignoreDependencies` in `knip.json`

**AI/ML dependency audit (D-10):** For each of these packages, trace the import chain:

- `@anthropic-ai/sdk` — check `backend/src/` for `import.*@anthropic`
- `openai` — check `backend/src/` for `import.*openai`
- `@langchain/core`, `@langchain/community`, `langchain` — check for `import.*langchain`
- `@mastra/core` — check for `import.*mastra`
- `@xenova/transformers` — check for `import.*xenova`

For each package:

- If real import exists in production code: KEEP, update to latest stable (per D-09)
- If only imported in test files or unused: REMOVE with `pnpm remove --filter <workspace> <pkg>`

**Step 3: Handle unused exports.**

For each unused export Knip reports:

- If the export is in a file that ONLY exports that symbol: delete the file
- If the export is one of several in a file: remove just the unused export
- Placeholder services for future phases (clearance middleware, MoU notifications, migration tooling per D-07): do NOT delete. Instead add a Knip ignore directive above the export:
  ```typescript
  // TODO: Implement real clearance-based access control (Phase 3: SEC-02)
  // knip-ignore -- placeholder for Phase 3 implementation
  export function checkClearanceLevel(requiredLevel: string) { ... }
  ```

**Step 4: Handle unused files.**

For each unused file Knip reports:

- Verify the file is genuinely unreachable (not dynamically imported via `React.lazy`, route config, or i18n loader)
- If genuinely unused: delete the file
- If dynamically imported: add the import source to `knip.json` entry points

Common false positives to watch for:

- Route files loaded by TanStack Router's `routeTree.gen.ts` — already listed as entry point
- i18n JSON files loaded by i18n config — may need `frontend/src/i18n/index.ts` as entry point
- Seed data files — may need `backend/src/seed.ts` as entry point
- Test files — add `**/*.test.{ts,tsx}` to project globs if tests are being analyzed

Update `knip.json` entry points and ignore patterns as needed to resolve false positives.

**Step 5: Handle unlisted dependencies.**

If Knip reports unlisted dependencies (used but not in package.json), install them in the correct workspace:

```bash
pnpm add --filter <workspace> <pkg>
```

**Step 6: Verify clean Knip report.**

```bash
pnpm exec knip --no-progress
```

Must exit with zero findings.

**Step 7: Verify build still passes after dependency removal.**

```bash
pnpm build
```

If build fails due to a removed dependency, add it back and investigate why Knip flagged it.

**Per D-08:** Scan and remove unused i18n translation keys from both EN and AR JSON files.

**Step 1: Identify all i18n JSON namespace files.**

```bash
find frontend/src/i18n/locales -name '*.json' | sort
```

**Step 2: For each namespace JSON file, extract all top-level and nested keys.**

For a JSON file like `frontend/src/i18n/locales/en/common.json`, extract the full key paths (e.g., `common.title`, `common.actions.save`).

**Step 3: Search the frontend source for each key.**

For each key, search `frontend/src/` (excluding the i18n locale files themselves) for references. Keys are typically used as:

- `t('namespace.key')` or `t('namespace.key.nested')`
- `t('key')` when the namespace is set via `useTranslation('namespace')`
- Sometimes as template literals: ``t(`${prefix}.${key}`)``

A key is UNUSED if:

- It does not appear in any `.ts` or `.tsx` file in `frontend/src/` (excluding `i18n/locales/`)
- It is not referenced via a dynamic key pattern (check for `t(\`...`)` patterns that could match)

**Step 4: Remove unused keys from BOTH EN and AR JSON files simultaneously.**

When removing a key from `en/common.json`, also remove the same key from `ar/common.json`. The two files must stay in sync.

**Step 5: Verify no runtime errors.**

After removal, verify the app still builds:

```bash
pnpm exec tsc --noEmit --project frontend/tsconfig.json
```

**IMPORTANT:** Be conservative. If a key MIGHT be used via dynamic pattern (e.g., `t(\`priority.${level}\`)`where level could be any value), keep ALL keys under that prefix. Only remove keys that are clearly orphaned with no dynamic references.
