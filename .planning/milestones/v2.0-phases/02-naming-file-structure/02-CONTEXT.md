## Phase Boundary

Enforce one predictable naming convention per layer across the entire monorepo. Every file, function, component, and service follows the convention for its layer. No directory reorganization or feature changes — naming only.

</domain>

<decisions>
## Implementation Decisions

### Component file convention

- **D-01:** All directories in `frontend/src/components/` use kebab-case (e.g., `activity-feed/`, `after-action/`, `ai/`, `analytics/`)
- **D-02:** Standalone `.tsx` files (e.g., `AfterActionForm.tsx`, `DossierCard.tsx`) move into their own kebab-case directories (e.g., `after-action-form/AfterActionForm.tsx`)
- **D-03:** No barrel exports (index.ts) in component directories — import directly from the file (e.g., `@/components/after-action-form/AfterActionForm`)
- **D-04:** Component files inside directories remain PascalCase (e.g., `AfterActionForm.tsx` inside `after-action-form/`)

### Hook naming

- **D-05:** All hook files use camelCase: `useAuth.ts`, `useCompliance.ts`, `useTasks.ts` — the ~18 kebab-case hooks (`use-compliance.ts`, `use-tasks.ts`) get renamed to camelCase
- **D-06:** Hook function names already follow `useX` convention — file names mirror the function name

### UI primitives

- **D-07:** `frontend/src/components/ui/` keeps kebab-case naming (`button.tsx`, `card.tsx`, `heroui-button.tsx`) — no changes needed, already consistent and matches shadcn/ui conventions

### Rename strategy

- **D-08:** Layer-by-layer execution: hooks first, then components, then cleanup/verification — each layer independently testable
- **D-09:** Use `git mv` for all renames to preserve blame history
- **D-10:** Naming only — no directory restructuring (don't move files between `pages/`, `routes/`, `services/`, `domains/`). Directory restructuring is a separate concern

### Enforcement

- **D-11:** Add ESLint filename rules (e.g., `eslint-plugin-filenames` or custom rules) to enforce: kebab-case for dirs/types/lib/ui, camelCase for hooks, PascalCase for component files inside dirs
- **D-12:** ESLint rules run in pre-commit hooks (already set up in Phase 1) — automated enforcement, not just documentation

### Claude's Discretion

- Exact ESLint plugin choice and rule configuration
- Order of files within each layer rename
- How to handle edge cases (e.g., `__tests__/` directories, `.d.ts` declaration files)
- Import path update tooling (manual find-replace vs automated script)

</decisions>

<code_context>

## Existing Code Insights

### Reusable Assets

- **Phase 1 pre-commit hooks**: ESLint + Prettier + build + Knip already run on commit — new ESLint filename rules plug directly into this
- **Knip** (dead code scanner): Will catch any broken imports after renames
- **TypeScript strict mode**: Compiler will flag broken imports immediately

### Established Patterns

- `frontend/src/types/` — consistently kebab-case with `.types.ts` suffix (good reference pattern)
- `frontend/src/lib/` — consistently kebab-case (good reference pattern)
- `frontend/src/components/ui/` — consistently kebab-case (shadcn/ui convention, already decided to keep)
- Backend (`api/`, `middleware/`, `services/`) — already kebab-case, minimal work needed

### Integration Points

- **~100+ import paths** will change when hooks and components are renamed
- **TanStack Router** route files in `frontend/src/routes/` — these are auto-generated by the router and may have their own naming requirements
- **`routeTree.gen.ts`** — auto-generated file, will regenerate after renames

### Current Inconsistencies (scope of work)

- `frontend/src/components/`: ~30 PascalCase files, ~25 kebab-case dirs, ~15 camelCase dirs
- `frontend/src/hooks/`: ~18 kebab-case (`use-x.ts`) vs ~40 camelCase (`useX.ts`)
- Backend is already mostly consistent — minimal renames expected
