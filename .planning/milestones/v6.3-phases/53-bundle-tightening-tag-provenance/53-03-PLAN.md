<threat_model>

## Trust Boundaries

| Boundary                              | Description                                                                                                          |
| ------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| SSH key → git tag                     | The SSH private key signs tag objects; possession of the key = ability to forge a `phase-NN-base` signed tag         |
| `allowed_signers` file → `git tag -v` | The verifier trusts the principal-to-key mappings in this file; tampering would let an attacker validate forged tags |
| local tag → origin tag                | Force-push to `origin` is auditable in the GitHub tag history; reflog on origin shows previous SHAs                  |

## STRIDE Threat Register

| Threat ID | Category                | Component                             | Disposition | Mitigation Plan                                                                                                                                                                                                                                                                                                         |
| --------- | ----------------------- | ------------------------------------- | ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| T-53-06   | Spoofing                | SSH signing key                       | mitigate    | D-06 reuses the existing GitHub-enrolled SSH key (already protected at the machine level by the user's existing OS keychain / ssh-agent posture); no new key material is generated that would expand the attack surface. Key compromise risk is the same as for repo push access, which is already accepted             |
| T-53-07   | Tampering               | `~/.ssh/allowed_signers`              | mitigate    | File is created with `chmod 600` (user-only read/write) per Task 2; sits in `~/.ssh/` alongside the private key with the same OS-level protections. Not committed to the repo, so a repo write does not let an attacker rewrite trust                                                                                   |
| T-53-08   | Tampering / Repudiation | Force-push of tag refs to origin      | mitigate    | D-10 documents force-push as the standard tag-re-issue idiom. GitHub stores previous tag SHAs in its tag history UI so the re-issue is auditable. Local re-issue captures the exact pre-existing SHA (D-08 step 1) so the tag target does not silently move — this is enforced by the Task 3 verify gate comparing SHAs |
| T-53-09   | Information disclosure  | `~/.ssh/allowed_signers` (public key) | accept      | The file contains only the SSH public key blob (already published to github.com when the key was enrolled) and the user's email (already in `git log` of every commit). No secret material exposed                                                                                                                      |
| T-53-10   | Tampering (supply)      | Package installs                      | n/a         | No `npm`/`pnpm`/`pip`/`cargo install` in this plan. The Phase 49 D-05 Package Legitimacy Gate does not apply; no `[ASSUMED]` / `[SUS]` packages introduced. Threat T-{phase}-SC is not present                                                                                                                          |
