<objective>
Phase 40 Gap-Closure G11: bring the Phase 40 EngagementsListPage to parity with the legacy view by fixing three independent root causes:

1. **Route mismatch** — the canonical path `/dossiers/engagements` still serves the legacy Feature 028 `<Table>` layout (an inline component defined in the route file). Re-wire it to render `@/pages/engagements/EngagementsListPage` (the Phase 40 ListPageShell-based view). This removes the route-discovery confusion that put Test 8 into `result: issue, severity: major`.

2. **Untranslated filter pill labels** — `EngagementsList.tsx` builds its FILTERS array with double-prefixed labelKeys (`'engagements.filter.all'`) AND looks them up under namespace `'engagements'`, so the resolver searches `engagements.engagements.filter.all` (which doesn't exist) and falls through to `defaultValue: f.value` — printing the raw lowercase enum value. Same shape bug on the search placeholder, the filter aria-label, the week heading, the row aria-label, and the load-more button text. Fix by stripping the redundant `engagements.` prefix from labelKeys / `t()` calls AND adding the matching keys to the bundled `engagements.json` (EN+AR).

3. **Zero rows render** despite 3 seeded confidential engagements in staging. Diagnosed in CONTEXT as RLS/clearance, but the actual root cause is a TYPE-STRING MISMATCH:
   - Canonical dossier type is `'engagement'` (per migration 20260202400000 trigger validation enum)
   - The Edge Function (`supabase/functions/engagement-dossiers/index.ts:366`) and the RPC `search_engagements_advanced` (migration 20260110000006:485) both filter on `d.type = 'engagement_dossier'`
   - The seeded data uses `type='engagement'`, so the WHERE clause excludes every row
   - Seeded engagements may also lack rows in the `engagement_dossiers` extension table that the RPC INNER JOINs

   Fix by (a) writing a migration that updates `search_engagements_advanced` to filter on `d.type='engagement'`, AND (b) backfilling `engagement_dossiers` rows for any existing dossiers WHERE type='engagement' that lack one. The Edge Function's `count()` query also needs the type-string fix; that change ships in the same migration via `CREATE OR REPLACE FUNCTION` plus a code edit to the Edge Function file IF the Edge Function is deployed from the repo. (Confirmed it is: `supabase/functions/engagement-dossiers/index.ts` is the source of the deployed function.)

Purpose: close 40-HUMAN-UAT.md Test 8 (`result: issue, severity: major`) and remove the dead-end legacy route.

Output: 5 file changes (route rewire + EngagementsList.tsx i18n cleanup + 2 i18n JSON edits + 1 SQL migration) and one Edge Function redeploy via Supabase MCP.
