# Quick Task 260601-gd5 — Notification walkthrough bugs #10 + #11

Two surgical code fixes, **no database migration**. Both root causes were
established with live-DB evidence in the debug sessions:

- `.planning/debug/resolved/digest-jobs-failed-query-users.md` (#10)
- `.planning/debug/resolved/notif-category-prefs-406.md` (#11)

## Task 1 — #10 backend digest scheduler

**File:** `backend/src/queues/digest-scheduler.ts`

- `processDailyDigests` (~L84): `.eq('digest_frequency', 'daily')` → `.eq('daily_digest_enabled', true)`
- `processWeeklyDigests` (~L200): `.eq('digest_frequency', 'weekly')` → `.eq('weekly_digest_enabled', true)` (keep `weekly_digest_day` filter)

**Why:** `digest_frequency` is not a column on `email_notification_preferences`.
PostgREST rejected the query with `42703` ("column does not exist"), which the
scheduler's catch block re-labeled as the misleading "failed to query users".
The real opt-in columns are the booleans `daily_digest_enabled` /
`weekly_digest_enabled` (both confirmed to exist on the live DB).

- **verify:** `pnpm --filter intake-backend run type-check` clean; corrected REST query → HTTP 200 (verified live).
- **done:** digest jobs no longer error on the user query; empty result = "no opted-in users", which is the correct non-crashing default.

## Task 2 — #11 frontend email-preferences hook

**File:** `frontend/src/hooks/useEmailNotifications.ts` (~L121)

- `.single()` → `.maybeSingle()`; drop the now-dead `PGRST116` branch; keep the return type nullable (`EmailNotificationPreferences | null`).

**Why:** `.single()` against the (empty) `email_notification_preferences` table
returns 406 (PGRST116, "0 rows"). `.maybeSingle()` returns `null` + HTTP 200,
and the page already falls back to defaults on null — the browser-visible 406 on
the Notification Preferences page disappears.

- **verify:** `pnpm --filter intake-frontend run type-check` clean.
- **done:** no 406 on the preferences page; defaults render when no row exists.
