# Quick Task 260624-d09: Serve the Mastra `/chat` route in agent-runtime prod

## Problem

The agent-runtime container serves **only `/health`**, not the AG-UI `/chat` SSE route
the copilot drawer POSTs to. Result: `POST /api/copilot/chat` â†’ **404** (`Cannot POST
/chat`, Express's signature), so the copilot UI cannot reach the runtime at all.

**Root cause (diagnosed live this session):** `Dockerfile.prod` CMD is `node dist/index.js`
â†’ the bootstrap in `src/index.ts` only `await import('./mastra/index.js')` to _construct_
the Mastra route graph, then starts a bare `express()` server exposing **just `GET /health`**.
The Mastra/Hono server that actually serves `/chat` (registered via
`registerCopilotKit({path:'/chat'})`) is **never started**. `@mastra/deployer` / the
`mastra` CLI are not in the image, and `createHonoServer` is internal/unexported.

This is why Phase-72 PROOF 5 (e2e) was perpetually "pending" â€” not a GPU/deploy gap, a
serving-wiring bug surfaced the first time the e2e was actually driven.

## Proven approach (verified in-container before coding)

`mastra.getServer()` returns `{ port, cors, apiRoutes }`. Mounting those `apiRoutes`
(`/chat` + `/health`) on a fresh Hono app with `c.set('mastra', mastra)` and serving via
`@hono/node-server` **works** â€” in-container probe returned `GET /health â†’ 200`,
`POST /chat â†’ 400` (handler executed; 400 only because the probe body was deliberately
malformed). No new server-adapter package, no `@mastra/deployer` needed.

- One deduped `hono@4.12.26` instance (so my Hono app and the handlers share context) â€”
  `@mastra/core@1.43.0` is exact-pinned, so declaring `hono`/`@hono/node-server` as direct
  deps at the resolved versions keeps a single instance. `shamefully-hoist=true` already
  flattens them.
