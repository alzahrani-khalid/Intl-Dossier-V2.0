# Intl-Dossier v7.0 — On-Prem Agentic-AI Architecture Research Dossier

**Status:** Decision-ready research synthesis
**Date:** 2026-06-13
**Author:** Lead architect (synthesis of 5-dimension verified research)
**Scope:** Make Intl-Dossier a state-of-the-art, **on-premise**, agentic-AI system where a CopilotKit/AG-UI surface is both the **primary interaction layer** (talk to it, it drives the dossiers) and an **assistant layer** on conventional screens — without violating data sovereignty, the clearance-based RLS model, or the Arabic-first/RTL mandate.

## 1. Executive summary — the single recommended architecture

Adopt **AG-UI Protocol as the non-negotiable wire contract**, render it with a **headless React client** (`assistant-ui` + `@assistant-ui/react-ag-ui`, or a thin custom `@ag-ui/client`) bound to the IntelDossier design tokens, and back it with a **TypeScript-native Mastra agent runtime** (already a dependency at `backend/src/ai/mastra-config.ts`) hosted in a **new `agent-runtime` Turborepo workspace** that emits AG-UI over SSE. Serve models **fully on-prem with vLLM** (OpenAI-compatible `/v1`), running a small fleet — **QCRI Fanar-2-27B-Instruct** (Apache-2.0, 32K context, native tool calling, native-Arabic reasoning) as the primary Arabic + agent brain, **Qwen3.5** (MoE 30B-A3B / dense 32B) for high-concurrency tool dispatch, and **SDAIA ALLaM-7B** as a sovereignty-signaling secondary for short Arabic tasks — with **llama.cpp/Ollama** as the identical-API modest/airgapped fallback. Keep retrieval in **pgvector** (no second datastore): a single `rag_chunks` table with **hybrid search (HNSW + tsvector/pg_trgm fused by RRF) + a local `bge-reranker-v2-m3` cross-encoder via TEI**, re-embedded once to **`bge-m3` at native 1024-dim halfvec** to end today's dimension chaos and the OpenAI dependency. The architectural **keystone is identity, not a guardrail bolt-on**: every agent DB read/write runs through a Supabase client carrying the **caller's JWT** so Postgres RLS enforces `sensitivity_level <= profiles.clearance_level` automatically — retiring the current `supabaseAdmin` (service-role) anti-pattern in `chat-assistant.ts`. **AnythingLLM is retired** from the critical path (wrapped as an MCP tool only during migration). Net: a TS-native, single-datastore, fully-local stack where the agent physically cannot read above the caller's clearance.

## 2. Per-layer recommendation

### 2.1 Agentic-UI / Frontend-protocol layer → **AG-UI contract + headless React client**

**Pick:** AG-UI Protocol (MIT, vendor-neutral wire standard) as the contract; **`assistant-ui` with `@assistant-ui/react-ag-ui`** (or a thin custom `@ag-ui/client`) as the React 19 client. Treat **CopilotKit** as a _selective accelerant_ — borrow its `@copilotkit/runtime` and `useCopilotReadable`/`useCopilotAction`/`useCoAgent` hook ergonomics where they beat build cost, but **render with your own headless components, not its chrome**.

**Why this and not CopilotKit's built-in chrome:**

- AG-UI is _just a wire format_ — ~20+ typed JSON events (lifecycle, text, tool-call, `STATE_SNAPSHOT`/`STATE_DELTA` JSON-Patch, reasoning, custom) over SSE/WS. It is emitted natively by LangGraph, **Mastra**, CrewAI, Pydantic AI, Agno, LlamaIndex, Google ADK, etc., so the on-prem agent framework becomes a swappable detail.
- The hard constraint is the **token-bound, no-shadcn, no-raw-hex, Arabic-first RTL** design system. `assistant-ui` and a custom `@ag-ui/client` are **headless** — every element is your component obeying `var(--*)`, `--row-h`, `1px solid var(--line)`, logical properties, `dir="rtl"` + Tajawal. They also talk **directly** to your AG-UI server via `HttpAgent` (no CopilotKit proxy hop).
- CopilotKit's built-in `CopilotSidebar`/`CopilotChat` is shadcn/Tailwind-flavored, restylable only via `--copilot-kit-*` CSS vars + per-slot overrides + a headless mode, and its built-in components **do not document RTL** — so for the primary surface it is a fight, not a fit.

**Verified correction applied:** It is _not_ true that "no library advertises first-class Arabic-RTL support" — **`assistant-ui` ships a dedicated RTL doc page**, its pre-styled components use logical Tailwind classes (`ms-*`, `pe-*`, `text-start`, `border-s`) that auto-flip under `dir=rtl` ("nothing to opt out of"), plus a `DirectionProvider`. The headless route is still the right preference for a strict token-only system, but on its actual merits (no foreign chrome to override), not on the false premise that every option dumps all RTL work on you.

**Rejected:** Vercel AI SDK UI is **not an AG-UI client** (would need a custom bridge) and its `streamUI` generative UI is RSC-oriented, which a TanStack-Router SPA does not use — relegate it to an _internal model/streaming detail_ on the server side.

### 2.2 Agent orchestration / runtime layer → **Mastra (TS-native), embedded, AG-UI first-party**

**Pick:** **Mastra** (Apache-2.0, TS-native, GA Jan 2026) as the agent runtime. It is the _only_ option that satisfies every hard constraint for this stack simultaneously:

- **First-party AG-UI emission** (`registerCopilotKit()` / `@ag-ui/mastra`) — no hand-written emitter to maintain; gives shared state + generative UI + HITL to the React surface.
- **TypeScript-native** — drops into the Turborepo with the same `tsconfig` strict, ESLint, Sentry, CI. **No polyglot Python service** to deploy/observe/secure/staff.
- **Fully local/on-prem** — runs against Ollama/llama.cpp/vLLM; persists threads/memory/workflow state on the **same Supabase Postgres+pgvector** via `@mastra/pg` (no new datastore).
- **Maintainable** — Zod-typed tools (matching existing validation), built-in `suspend()`/`resume()` HITL, native MCP (the existing `anythingllm` MCP server plugs in), OpenTelemetry tracing out of the box.
- **Already in the repo** — `backend/src/ai/mastra-config.ts` (`defineAgent`/`createAgentTools`), `chat-assistant.ts`, `intake-linker.ts`. Lift, don't rewrite.

**Polyglot flag (decisive against the Python field):** LangGraph, CrewAI, Pydantic AI, LlamaIndex Workflows, and Google ADK would each force a **second Python runtime** alongside Node/Express — doubling deploy/observability/security/on-call surface for capabilities a TS-native runtime already covers. **LangGraph is additionally disqualified for on-prem-first**: its self-hosted _production_ server (`langgraph-api`) is **Elastic-License-2.0** and needs a paid Enterprise key — a direct sovereignty/licensing blocker. Pydantic AI's headline (type safety) is redundant given TS-strict + Zod.

**Documented fallback:** if Mastra's opinionated workflow/memory abstractions prove constraining during a spike, fall back to a **thin custom TS orchestrator over the Vercel AI SDK v6 `ToolLoopAgent`** + the `@ag-ui/vercel-ai-sdk` bridge — same stack, more control, but you then re-own persistence, HITL, and observability that Mastra ships.

**Non-negotiable rule regardless of framework:** every tool's `execute()` calls Supabase using the **end-user JWT**, never service-role (see §2.5).

### 2.4 Retrieval / RAG / Vector layer → **KEEP pgvector; hybrid + local rerank; re-embed to bge-m3 1024-dim**

**Pick:** **Keep pgvector — do not introduce a dedicated vector DB.** At thousands–low-millions of clearance-gated chunks, pgvector is sufficient _and_ strictly more maintainable: one datastore, one auth model, one backup, and — decisively — **the same clearance RLS that governs dossiers gates retrieved chunks** when the chunk table carries/joins `sensitivity_level` and search runs `SECURITY INVOKER`. A separate vector DB (Qdrant/Weaviate/Milvus/LanceDB) would force you to **re-implement and continuously sync** the clearance model outside Postgres — turning the security keystone into a drift-prone second copy. There is no scale justification below ~10M vectors.

**Verified corrections applied (do not repeat the refuted figures as fact):**

- The "single-digit ms at 1M–5M" latency claim is **wrong**. pgvector HNSW is ~**p95 8–15 ms at 1M** (1536-dim), ~**20–40 ms** (halfvec, optimistic) up to **80–140 ms** at 5M single-instance; single-digit ms (2–5 ms) is only ~100K vectors. The **hybrid+RRF precision lift (~62% → ~84%+) and reranker (+3–7% nDCG) figures hold** (single-source blog anecdotes, not a controlled benchmark — treat as directional). Sufficiency-under-~10M and the dedicated-DB crossover (~10–50M, clearly past ~50M) are **correct**; one source puts the comfortable single-node boundary nearer ~2M (index-build/VACUUM contention), so **5M is the optimistic top end**.
- RLS-gated retrieval is **not unique to pgvector** — Milvus (row-level RBAC), Weaviate (multi-tenancy + RBAC), Pinecone (namespaces), Qdrant (server-side payload filters) all do access-gated retrieval. The Supabase "RAG with Permissions" guide attributes the capability to **Postgres RLS**, not pgvector specifically. The pgvector win here is **single-datastore + zero-sync of the existing clearance model**, not exclusivity.
- **`bge-m3` license is MIT, not Apache-2.0** (factual fix). It _is_ native 1024-dim (max seq 8192), _is_ this repo's local ONNX dev embedder (`backend/src/ai/embeddings-service.ts` → `Xenova/bge-m3`, config default 1024), and led the tested set for Arabic RAG (RAGAS 70.99 vs e5-large 70.31) — but by a **0.68-point margin** (a narrow lead, not a categorical win). The cited Precision ~74 / Recall ~72 are **not substantiated** by the source; drop them.

**Target retrieval architecture (on-prem):**

1. **One `rag_chunks` table** (`content`, `content_tsv_en`/`content_tsv_ar` generated tsvector, `embedding halfvec(1024)`, **denormalized `sensitivity_level`** kept synced from the parent dossier by trigger). Enable RLS with the existing predicate; run search **`SECURITY INVOKER`** (drop the `SECURITY DEFINER` retrieval pattern — that is today's leakage gap).
2. **Hybrid in one SQL function:** pgvector HNSW (`halfvec_cosine_ops`) + `ts_rank_cd` (reuse existing EN/AR GIN indexes) + optional `pg_trgm`, fused with **RRF (k≈60)**. Enable **pgvector 0.8 iterative index scans** (`hnsw.iterative_scan='relaxed_order'`, `hnsw.max_scan_tuples`) so the RLS post-filter doesn't collapse recall for high-clearance users. Consider partial HNSW indexes / partition-by-sensitivity for the most selective tiers.
3. **Rerank top ~50 RLS-passing candidates** with local **`bge-reranker-v2-m3` via TEI**, keep top ~8–10 for the LLM. **RLS must run before rerank** — the cross-encoder cannot enforce clearance.
4. **Serve `bge-m3` (embeddings) + `bge-reranker-v2-m3` (rerank) from one on-prem TEI container** — the _only_ new service. AnythingLLM leaves the retrieval critical path.

**Dimensions — move off 1536 to a single LOCAL 1024-dim model and re-embed once.** The store is inconsistent today (1536 vs 1024 vs an edge function that **pad/truncates to 1536, corrupting cosine geometry**). `bge-m3` 1024 is the best open Arabic+English model and already the dev path; halfvec(1024) halves storage; this ends the OpenAI dependency. **Remove the `normalizeEmbedding(...,1536)` pad/truncate in `supabase/functions/search-semantic/index.ts`** — it silently corrupts vectors. A one-time re-embed is unavoidable when changing models and is the moment to consolidate every embedding into `rag_chunks` with proper clearance RLS.

**⚠️ Reconcile FIRST — clearance-scale mismatch:** `profiles.clearance_level` is **1–4**, `get_user_clearance_level()` returns **1–3** from `user_roles`, and `dossiers.sensitivity_level` is **low/medium/high**. **Pick one canonical scale before wiring chunk RLS** or gating will be inconsistent.

### 2.5 Security topology → **the clearance-RLS-via-JWT keystone (the crux)**

> This is the single most important decision in the whole dossier. An on-prem diplomatic AI is only as safe as its weakest data path. The agent must be **incapable** of reading above the caller's clearance — by _construction_, not by remembering to check.

**Pick:** **JWT propagation — a user-scoped Supabase client, never service-role.** The user's Supabase access token rides from the React provider's `properties.authorization`, through the AG-UI SSE POST, into the agent's `forwardedProps`. Every data-touching Mastra tool builds a per-request client:

```ts
createClient(SUPABASE_URL, SUPABASE_ANON_KEY, {
  global: { headers: { Authorization: `Bearer ${jwt}` } },
  auth: { persistSession: false, autoRefreshToken: false },
})
```

Postgres sets `auth.uid()`/`auth.jwt()` from the token, so existing RLS — including `sensitivity_level <= (SELECT clearance_level FROM profiles WHERE id = auth.uid())` — executes on every agent query with **zero app-layer authz code**. This is the **exact idiom already proven in ~20 edge functions** (`positions-versions-compare:101`, `attachments-list:96`, `interaction-notes-list:37`).

**The anti-pattern to retire:** `backend/src/ai/agents/chat-assistant.ts` uses **`supabaseAdmin` (service-role) across all queries (lines 158, 171, 202, 235, 269, 307, 338)** — this **bypasses RLS entirely**. A prompt-injection or a logic bug there can exfiltrate above-clearance dossiers. Service-role is reserved **strictly** for cron/no-user system paths (signal ingestion, scheduled digests) with **explicit** app-layer authz — never shared with the interactive agent.

**Why JWT-propagation is the right keystone:**

- Enforcement is automatic and **cannot be forgotten per-tool** — the DB is the single enforcement point; new intelligence tables inherit it free if they ship with the same RLS.
- **Prompt injection cannot escalate data access** — a hijacked agent still only sees what the user could see (injection can misuse _intent_, not exceed _clearance_).
- Audit trail is **honest** — queries run as the real user, not a god account.

**Caveat to engineer:** Supabase access tokens (~1h) can expire mid-run. For long agent runs, pass the refresh token server-side or bound runs under the TTL; set `persistSession:false, autoRefreshToken:false` on per-request clients.

**Defense-in-depth (layered, since feeds are untrusted content):**

1. **RLS-as-floor** (worst-case injection bounded by the user's own clearance).
2. **Least-privilege typed tools** — each does one scoped op; **never a generic `execute_sql`**.
3. **Human-in-the-loop on ALL writes** — every state-changing dossier action is a frontend action with **`renderAndWaitForResponse`** rendering a bilingual, token-bound confirmation card; the write commits only after `respond()`, and **under the user's JWT** (RLS still applies).
4. **Intent-vs-action guardrail** — evaluate each proposed tool call against the user's original task _without_ untrusted context.
5. **Quarantine / dual-LLM** for untrusted ingested feed content (treat v7.0 signal feeds as untrusted).
6. **Full trace logging** + input/output filtering.

**Runtime topology:** a **new `agent-runtime` TypeScript workspace** (4th alongside `backend`/`frontend`/`shared`): Express + `@copilotkit/runtime` (or `@ag-ui/*` server SDK) hosting the Mastra agent, terminating AG-UI SSE, calling the local model over OpenAI-compatible HTTP, exposing dossier ops as narrow typed tools (optionally re-exposed as an MCP server). Deploys as its own container in `docker-compose.prod.yml`. This keeps long SSE streams + model latency **off the transactional Express API** (and off the port-5000 AirPlay conflict — use a distinct port). Use the **AG-UI/self-hosted-runtime path, not A2A** — the header-forwarding bug (#3170) affects A2A only.

**On-prem observability/eval:** **Langfuse self-host** (tracing, cost, prompt registry/versioning — satisfies the v7.0 "prompt registry + audit trail" requirement directly) + **Arize Phoenix** (offline eval, bilingual rubrics), both fed by **OpenLLMetry/OTel** so instrumentation is vendor-neutral. Zero telemetry egress (adds ClickHouse to the footprint for Langfuse).

## 3. End-to-end architecture options (scored)

Scoring: 1–5 (5 best). Axes are the user's four — **Scalable / Maintainable / Efficient / Effective** — plus **On-prem fidelity** and **Time-to-value**.

### Option A — **TS-native: Mastra + AG-UI(headless) + vLLM(Fanar/Qwen) + pgvector hybrid** ⭐ RECOMMENDED

Mastra runtime in `agent-runtime`, AG-UI over SSE → `assistant-ui`/custom headless client, vLLM serving the Saudi-gov fleet, pgvector hybrid + TEI rerank, JWT-propagation keystone, Langfuse/Phoenix.

| Axis             | Score | Note                                                                                                                |
| ---------------- | ----- | ------------------------------------------------------------------------------------------------------------------- |
| Scalable         | 4     | vLLM continuous batching + MoE for concurrency; pgvector fine to ~5M chunks; agent isolated for independent scaling |
| Maintainable     | 5     | One language (TS), one datastore (Supabase), reuses existing Mastra + edge-fn idioms; no polyglot service           |
| Efficient        | 4     | Headless client = smaller bundle; halfvec 1024 = half the index; MoE = ~3B active cost; one TEI container           |
| Effective        | 5     | First-party AG-UI shared state + generative UI + HITL; Fanar native-Arabic reasoning; best Arabic RAG               |
| On-prem fidelity | 5     | Everything local & permissively licensed (Apache/MIT); no data leaves the building                                  |
| Time-to-value    | 4     | Mastra already imported; idioms already proven; main cost is re-embed + agent-runtime stand-up                      |

**Why #1:** the _only_ option that satisfies every hard constraint at once — TS-native (no Python service), fully local & permissive, AG-UI-native, single-datastore RLS keystone, Arabic-first models. It also has the **shortest path** because Mastra and the JWT-bound client idiom already exist in the repo.

### Option B — **LangGraph (Python agent service) + CopilotKit + vLLM + pgvector**

Best-in-class graph orchestration and partner-grade CopilotKit integration.

| Axis             | Score | Note                                                                                                                                          |
| ---------------- | ----- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| Scalable         | 4     | Mature stateful graphs, Postgres checkpointer                                                                                                 |
| Maintainable     | 2     | **Second Python runtime** = polyglot deploy/observability/security surface in a TS shop                                                       |
| Efficient        | 3     | Capable but heavier conceptual + operational weight than the workload needs                                                                   |
| Effective        | 5     | Best for complex branching/checkpointed flows; excellent HITL                                                                                 |
| On-prem fidelity | 2     | **`langgraph-api` production server is Elastic-License-2.0** — needs a paid Enterprise key to self-host: a real sovereignty/licensing blocker |
| Time-to-value    | 2     | New Python service + licensing negotiation before production                                                                                  |

**Verdict:** disqualified by the **EL-2.0 production-server license** + polyglot cost. Reconsider only if a future workload genuinely needs LangGraph-class graph semantics _and_ the Enterprise license is acquired.

### Option C — **Lighter/faster first slice: thin custom AI-SDK orchestrator + CopilotKit-hooks + Ollama + pgvector(interim)**

A minimal Express route running Vercel AI SDK `ToolLoopAgent` + `@ag-ui/vercel-ai-sdk` bridge, CopilotKit hooks on conventional pages, Ollama for a demo box, pgvector with the **interim RLS fix** (no re-embed yet).

| Axis             | Score | Note                                                                       |
| ---------------- | ----- | -------------------------------------------------------------------------- |
| Scalable         | 2     | Ollama is ~9× lower throughput than vLLM; not multi-user production        |
| Maintainable     | 3     | You own persistence/HITL/observability the framework would ship            |
| Efficient        | 3     | Minimal deps, but re-builds what Mastra gives free                         |
| Effective        | 3     | Works for a demo; weaker shared-state/HITL story without framework support |
| On-prem fidelity | 5     | Fully local                                                                |
| Time-to-value    | 5     | **Fastest to a working demo** — smallest surface area                      |

**Verdict:** an excellent **Phase-D spike / demo path**, not the production target. Its real value is de-risking the AG-UI loop _before_ the GPU server and re-embed land — then graduate to Option A (same stack, same OpenAI-compatible model URL, so the swap is config-level).

**Recommendation: Option A.** It is the only choice that holds all hard constraints simultaneously and has the shortest runway because its core pieces (Mastra, JWT-bound client idiom, local model providers) already exist in the codebase. Use **Option C as the throwaway first slice** to validate the loop, then converge on A.

## 4. Integration plan with the existing stack

### 4.1 Where the agent runtime lives

- New **`agent-runtime`** workspace added to `pnpm-workspace.yaml` (currently `backend`/`frontend`/`shared`) + a `turbo.json` pipeline entry; share types via `shared/`.
- **Lift, don't rewrite:** move `backend/src/ai/mastra-config.ts` (`defineAgent`/`createAgentTools`) and `backend/src/ai/llm-router.ts` into `agent-runtime`.
- Mount `@copilotkit/runtime` (or `@ag-ui/*` server SDK) on its Express app; emit AG-UI over SSE; expose dossier ops as narrow typed Mastra tools.
- New service in `deploy/docker-compose.prod.yml` next to postgres/redis; CORS via `ALLOWED_ORIGINS` **secret** (not hardcoded — per prior edge-fn lesson); distinct PORT (avoid 5000/AirPlay).
- Install: `@ag-ui/core`, `@ag-ui/client`, `@ag-ui/mastra` (or `@assistant-ui/react-ag-ui` on the client), `@mastra/core`, `@mastra/pg`, `@copilotkit/runtime` (if used).

### 4.2 How AG-UI streams to React 19

- Mount the headless client inside the **`_protected`** route tree (only authenticated users reach it).
- Provider carries `{ authorization: 'Bearer ' + session.access_token, user_id, language: i18n.language }`. The headless `HttpAgent`/runtime parses AG-UI events (text, tool-call, `STATE_SNAPSHOT`/`STATE_DELTA`, reasoning, HITL) and renders them with **your token-bound components**.
- The existing AI Assistant surface (`frontend/src/pages/word-assistant/` + `routes/_protected/word-assistant.tsx`) and Cmd+K become the **primary conversational surface**; sprinkle `useCopilotReadable`/`useCopilotAction`-style hooks (or `assistant-ui` equivalents) on dossier-detail and network panels for **in-context assist**.

### 4.3 How tools map to the dossier action model + MCP

- **Read tools:** gated hybrid RAG RPC, dossier/work-item lookups — all via `getUserClient(jwt)`, never `supabaseAdmin`.
- **Write tools:** create/update/link `work_item_dossiers`, generate briefs, dismiss-signal, publish-digest — each wrapped in **`renderAndWaitForResponse`** HITL and mapped to Mastra `suspend()`/`resume()`.
- **Generative UI:** map AG-UI tool-call/generative events to **`DossierContextBadge`, `UniversalDossierCard`, KPI/list** patterns from the design system — _not_ generic chat bubbles. This is how the copilot "drives dossiers" on-brand.
- **MCP:** register the existing `anythingllm` MCP server (`.mcp.json` lists `anythingllm`, `filesystem`, `docker`) via Mastra's native MCP client; optionally re-expose dossier tools as an MCP server.
- **State sync:** reconcile `STATE_DELTA` into the **TanStack Query cache** so conventional surfaces and the copilot stay consistent.

### 4.4 Arabic-RTL theming on the copilot surface

- Headless client → every element is your component: `var(--*)`, `bg-surface`/`text-ink`/`border-line`, `1px solid var(--line)`, **no card shadows, no raw hex, no shadcn defaults**.
- Set `dir={isRTL?'rtl':'ltr'}` on the chat container; logical properties only (`ms-`/`me-`/`ps-`/`pe-`/`text-start`); Tajawal for Arabic; isolate LTR/code snippets with `dir="ltr"`; mirror directional icons per project RTL rules.
- Agent **answers in `i18n.language`** — tool descriptions, system prompts, and any agent-authored copy produced in the active locale. `chat-assistant.ts` already has a `language?: 'en'|'ar'` field (line 41) to thread through.
- **i18n trap:** register any new namespace in `src/i18n/index.ts` (static bundle — unregistered namespaces silently fall back to English, breaking Arabic). Language persists under `localStorage['id.locale']`.

### 4.5 What happens to AnythingLLM → **RETIRE (wrap during migration)**

- AnythingLLM is **explicitly replaceable**, not a constraint. **Retire it from the critical path.**
- During migration, **wrap** its workspace RAG as an MCP tool so it stays reachable while the AI Assistant surface, search suggestions, and dashboard digest move onto Mastra tools **one at a time** (behind a feature flag), verifying clearance-RLS + bilingual output at each step.
- Final state: retrieval runs directly over the gated pgvector hybrid RPC + TEI; the `anythingllm` provider/service and MCP server are decommissioned.

## 5. Phasing — composing with the v7.0 Intelligence Engine seed

**The decisive call: DATA LAYER FIRST, then interleave the agent.** Wiring CopilotKit over today's stub digest and ad-hoc signals produces a _demo, not a product_ — the agent is only as good as the substance it can read and act on. Building the intelligence tables with clearance-RLS **up front** means the JWT-propagation agent needs **zero bespoke authz** to honor classification on the new tables. The v7.0 seed shows the schema groundwork is partly landed (`intelligence_event`/`intelligence_digest` enum from v6.3) but signals/digest pipeline are still stub-level.

| Phase                           | Deliverable                                                                                                                                              | Notes                                                                                                                                                           |
| ------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **A — Data: signals**           | `intelligence_signal` table + signal→dossier polymorphic links + **clearance-RLS** + ingestion (manual/AI)                                               | Mirror `work_item_dossiers` + its `idx_..._unique_active` partial index; reuse v6.3 `signal_source_type` enum. **Reconcile the clearance-scale mismatch here.** |
| **B — Data: digests/alerts**    | Digest pipeline + alert thresholds + subscriber model                                                                                                    | Cron uses service-role with explicit app-layer authz (carve-out from the interactive agent)                                                                     |
| **C — Data: graph**             | Analytic-graph RPCs (who-sits-on-which-forum, engagement chains)                                                                                         | `SECURITY INVOKER`, clearance-aware                                                                                                                             |
| **D — Platform: agent + reads** | `agent-runtime` + **JWT propagation** + read tools over A–C; **retire `supabaseAdmin` in `chat-assistant.ts`**; **re-embed to bge-m3 1024 + TEI rerank** | First user-visible AI. Start with **Option C thin slice** to validate the AG-UI loop, then converge to Mastra/Option A                                          |
| **E — Platform: writes**        | Write tools with **`renderAndWaitForResponse`** HITL; generative-UI dossier cards                                                                        | Every state change gated + bilingual + token-bound                                                                                                              |
| **F — Eval gate**               | Langfuse + Phoenix; **bilingual (EN/AR) eval rubrics** wired as a CI gate                                                                                | Briefing/correlation quality regression-tested                                                                                                                  |

A–C are **data-first**; D onward **interleaves** — once D lands, each subsequent data capability ships with its agent tool in the same phase (interleaved tail). This avoids both "agent with nothing to say" and "data with no consumer."

**Parallelizable in advance (no data dependency):** stand up vLLM + the model fleet, the TEI container, and the re-embed job design while A–C are built — so D has the model/retrieval substrate ready the moment the data exists.

## 6. Risks & open decisions (need your product call)

1. **Hardware / GPU budget.** Fanar-2-27B at FP8/AWQ wants one 48–80GB GPU; Qwen3.5 dense 32B similar; MoE variants trade VRAM for active-compute. _Decision:_ how many GPUs, what class, and is there a datacenter GPU at all — or do we ship the llama.cpp/Ollama fallback for v1? (Determines whether Option A or C is the _production_ target on day one.)
2. **Model fleet: Fanar-2 vs Qwen3.5 as the default brain vs ALLaM optics.** Fanar (Qatari-origin, best Arabic) vs Qwen (Chinese-origin, best agentic) vs ALLaM (Saudi national model, but 4K/no-tools). _Decision:_ the political/optics weighting for the Saudi-gov customer — is "uses the Saudi national model" a procurement requirement (forcing ALLaM into a visible role despite its ceiling), or is "best Arabic capability, fully on-prem" sufficient? All run local, so no data leaves the building regardless.
3. **Re-embed timing.** Moving 1536→1024 (bge-m3) is a one-time job that consolidates embeddings into `rag_chunks` with proper RLS. _Decision:_ schedule the re-embed now (clean) or ship the **interim mandatory fix** first (stop pad/truncate-to-1536, add clearance-RLS to current vector search) and re-embed later? Plus: **which canonical clearance scale** (1–4 vs 1–3 vs low/med/high) before chunk RLS is wired.
4. **Feed ingestion = untrusted content.** v7.0 signal feeds are content the agent will read — the indirect-injection surface. _Decision:_ how much quarantine/dual-LLM rigor (latency cost) vs throughput; and which feed sources are in scope for v1.
5. **Alert channels.** Digest/alert delivery — in-app only, email, or external? _Decision:_ affects the subscriber model in Phase B and any egress posture (email leaves the building).
6. **How much of v7.0 is platform vs feature.** This dossier treats A–C (signals/digests/graph) as _platform data the agent consumes_. _Decision:_ are signals/digests _also_ standalone user-facing features with their own UI (independent of the copilot), or purely the agent's substrate? This sets whether B/C get dedicated conventional surfaces or only agent tools.
7. **CopilotKit hooks vs pure headless.** Borrowing `@copilotkit/runtime` + hooks accelerates the "drives-dossiers" ergonomics but adds the proxy hop and dependency weight. _Decision:_ adopt CopilotKit runtime/hooks (render with your own components) for speed, or stay pure `assistant-ui`/custom `@ag-ui/client` for minimal surface?
