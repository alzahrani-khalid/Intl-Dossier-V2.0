## 1. What v7.0 is

Turn dossiers from passive records into a **state-of-the-art, fully on-prem, Arabic-first intelligence layer** experienced through **both** conventional analyst surfaces (signal triage, digests, analytic graph) **and** an agentic copilot (CopilotKit/AG-UI) that can read and act on that layer — with the agent **incapable by construction** of reading above the caller's clearance.

**Core value reaffirmed:** unified intelligence management for diplomatic operations — every relationship, commitment, and signal tracked in one secure, bilingual, sovereign platform.

## 3. Remediation-first (foundational pre-work — blocks correct RLS + retrieval)

Before any new intelligence tables are wired, three existing-code issues the research surfaced must be fixed, because everything downstream inherits them:

1. **Reconcile the clearance scale to one canonical model.** Today `profiles.clearance_level` is 1–4, `get_user_clearance_level()` returns 1–3, `dossiers.sensitivity_level` is low/med/high. Pick one scale; all new RLS keys off it. **Without this, chunk- and signal-level clearance gating is inconsistent.**
2. **Fix embedding corruption.** `supabase/functions/search-semantic/index.ts` pad/truncates vectors to 1536-dim (corrupts cosine geometry); the store is dimension-inconsistent. Stop the pad/truncate; converge on `bge-m3` 1024-dim.
3. **Retire the `supabaseAdmin` RLS-bypass** in `backend/src/ai/agents/chat-assistant.ts` (service-role across all reads) — move to JWT-scoped clients (the keystone). This is a live data-exposure path, fix it regardless of the rest.

## 5. Cross-cutting guarantees (apply to every phase)

- **Security:** every interactive agent DB op runs under the user JWT (RLS keystone); every write is HITL; service-role only on cron/no-user paths with explicit authz; feed-ingested content treated as untrusted (quarantine posture if/when feeds are added — feeds themselves are **out of v7.0 scope**, deferred to v7.1).
- **Bilingual/RTL:** every new surface + agent-authored copy in `i18n.language`; new i18n namespaces registered in `src/i18n/index.ts` (unregistered → silent English fallback); `dir="rtl"` + Tajawal + logical properties; design tokens only (no shadcn defaults, no raw hex, no card shadows).
- **On-prem fidelity:** no data egress; all models/embedders/rerankers/observability self-hosted and permissively licensed (Gemma's custom license is the one accepted exception, recorded above).
- **GSD discipline:** each phase = research → (UI-spec where applicable) → plan → check → execute → self-run live UAT on staging (seed/observe/restore, EN+AR) → code review → verify → PR → merge on green.

---

## 6. Deferred to v7.1+ (explicitly out of v7.0)

- **External feed ingestion** (RSS/public APIs/scheduled pulls) — the untrusted-content/indirect-injection surface; v7.0 signals are manual + AI-surfaced only.
- Multi-GPU/horizontal model-serving scale-out beyond single-server (revisit if concurrency demands).
- Advanced quarantine/dual-LLM rigor (lands with feeds in v7.1).

---

## 7. Open risk register (tracked, not blocking)

- **GPU capacity** — design targets a single ~16–24GB card (Gemma 4 12B + QAT); if a larger GPU lands, the eval may promote a 27–32B Arabic model. (Inferred from the Gemma-12B lean; confirm the actual datacenter GPU before Phase 72.)
- **Gemma Arabic adequacy** — gated by the Phase-68 eval harness; swap is config-level if it underperforms.
- **CopilotKit chat-shell air-gap** — resolved empirically in the Phase-72 Option-C spike (CopilotKit-restyled shell vs `assistant-ui`/`@ag-ui/client`).
- **Email/external egress posture** — alert delivery via on-prem SMTP + approved outbound webhook; confirm the sanctioned external endpoints with the customer before Phase 70.
